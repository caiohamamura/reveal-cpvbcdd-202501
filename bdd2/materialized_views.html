O Problema: Consultas Demoradas

  • Relatórios complexos que levam 8-10 segundos
  • Consultas executadas várias vezes por dia
  • Impacto na experiência do usuário
    
    SELECT p.product_name,
           COUNT(*) as total_vendido,
           SUM(quantity * unit_price) as valor_total
    FROM order_details od
    JOIN products p ON od.product_id = p.product_id
    JOIN orders o ON od.order_id = o.order_id
    WHERE o.order_date >= CURRENT_DATE - 30
    GROUP BY p.product_name
    ORDER BY valor_total DESC;

⏱️ Tempo: 8-10 segundos para executar

1. Criando a Materialized View


    -- Cria a MV com os dados carregados
    CREATE MATERIALIZED VIEW mv_relatorio_vendas AS
    SELECT 
        p.product_name,
        c.category_name,
        s.company_name AS supplier,
        SUM(od.quantity) AS total_vendido,
        SUM(od.quantity * od.unit_price) AS valor_total,
        COUNT(DISTINCT o.order_id) AS pedidos_unicos,
        ROUND(AVG(od.quantity * od.unit_price), 2) AS ticket_medio,
        CURRENT_DATE AS data_atualizacao
    FROM order_details od
    JOIN products p ON od.product_id = p.product_id
    JOIN categories c ON p.category_id = c.category_id
    JOIN suppliers s ON p.supplier_id = s.supplier_id
    JOIN orders o ON od.order_id = o.order_id
    WHERE o.order_date >= CURRENT_DATE - INTERVAL '30 days'
    GROUP BY p.product_name, c.category_name, s.company_name
    ORDER BY valor_total DESC
    WITH DATA;

WITH DATA: Carrega os dados imediatamente

2. Criando Índices para Performance


    -- Índice para ordenação por valor total
    CREATE INDEX idx_mv_valor_total 
    ON mv_relatorio_vendas_completo(valor_total DESC);
    
    -- Índice para filtro por categoria
    CREATE INDEX idx_mv_categoria 
    ON mv_relatorio_vendas_completo(category_name);
    
    -- Índice para filtro por fornecedor
    CREATE INDEX idx_mv_fornecedor 
    ON mv_relatorio_vendas_completo(supplier);

⚡ Índices aceleram buscas e ordenações

Comparação de Performance

View Regular


-- Executa a JOIN toda vez
CREATE VIEW vw_vendas AS
SELECT ... FROM ... WHERE ...
JOIN ...

-- Tempo de execução
SELECT * FROM vw_vendas;
-- ⏱️ 8-10 segundos

Materialized View


-- Resultados pré-calcados
CREATE MATERIALIZED VIEW mv_vendas AS
SELECT ...
WITH DATA;

-- Cria índices
CREATE INDEX idx_valor ON mv_vendas(valor_total);

-- Tempo de execução
SELECT * FROM mv_vendas;
-- ⏱️ < 0.1 segundo

Ganho de Performance: 80x

3. Atualizando a Materialized View


-- Opção 1: Refresh completo (bloqueante)
REFRESH MATERIALIZED VIEW mv_relatorio_vendas;

-- Opção 2: Refresh concorrente (não bloqueante)
REFRESH MATERIALIZED VIEW CONCURRENTLY mv_relatorio_vendas;
  • Refresh completo: Reprocessa tudo, mais rápido mas bloqueia
  • Refresh concorrente: Não bloqueia, mas precisa de índice único

Quando Usar Materialized Views?

  • ✓ Relatórios executados frequentemente
  • ✓ Dados que mudam pouco (estáveis)
  • ✓ Agregações complexas (SUM, AVG, COUNT)
  • ✓ Consultas com filtros repetitivos
  • ✗ Dados que mudam constantemente
  • ✗ Consultas que já são rápidas
  • ✗ Quando storage é limitado
## Trade-offs: Materialized Views **Performance vs Atualização** - **Prós**: - Consultas muito mais rápidas (< 0.1s vs 8-10s) - Reduz carga de processamento em queries complexas - Melhora a experiência do usuário - **Contras**: - Uso adicional de storage - Necessidade de manter dados atualizados - Complexidade de gerenciamento - Trade-off: performance de escrita vs leitura **Dica**: Use para relatórios onde a performance de leitura é mais importante que atualização em tempo real.
