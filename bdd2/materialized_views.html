📝 View Regular — Como funciona

Uma View é apenas uma consulta nomeada — ela não armazena dados.


CREATE VIEW vw_vendas_categoria AS
SELECT c.category_name,
       SUM(od.unit_price * od.quantity) AS valor_total,
       COUNT(DISTINCT o.order_id) AS total_pedidos
FROM order_details od
JOIN products p ON od.product_id = p.product_id
JOIN categories c ON p.category_id = c.category_id
JOIN orders o ON od.order_id = o.order_id
GROUP BY c.category_name;

⚠️ Toda vez que alguém faz SELECT * FROM vw_vendas_categoria, a query inteira roda de novo.

🗄️ Materialized View — Como funciona

Uma Materialized View executa a query uma vez e armazena o resultado fisicamente.


CREATE MATERIALIZED VIEW mv_vendas_categoria AS
SELECT c.category_name,
       SUM(od.unit_price * od.quantity) AS valor_total,
       COUNT(DISTINCT o.order_id) AS total_pedidos
FROM order_details od
JOIN products p ON od.product_id = p.product_id
JOIN categories c ON p.category_id = c.category_id
JOIN orders o ON od.order_id = o.order_id
GROUP BY c.category_name
WITH DATA;

✅ SELECT * FROM mv_vendas_categoria lê os dados pré-computados — como uma tabela comum.

🔄 Atualizando a MV: REFRESH

⚠️ Lembrete: A MV não atualiza sozinha. Você precisa disparar o REFRESH.


-- Atualização completa (bloqueia leituras durante o refresh)
REFRESH MATERIALIZED VIEW mv_vendas_categoria;

⛔ Problema: Durante o REFRESH, ninguém consegue ler a MV!

🔄 REFRESH CONCURRENTLY


-- Passo 1: Criar índice único (OBRIGATÓRIO)
CREATE UNIQUE INDEX idx_mv_vendas_cat_unique
  ON mv_vendas_categoria(category_name);

-- Passo 2: Refresh sem bloquear leituras
REFRESH MATERIALIZED VIEW CONCURRENTLY mv_vendas_categoria;

🚀 Índices em Materialized Views

Uma MV pode ter seus próprios índices, assim como uma tabela.


-- Índice para consultas por categoria
CREATE INDEX idx_mv_categoria
  ON mv_vendas_categoria(category_name);

-- Índice para ordenação por valor
CREATE INDEX idx_mv_valor
  ON mv_vendas_categoria(valor_total DESC);

-- Índice único (para REFRESH CONCURRENTLY)
CREATE UNIQUE INDEX idx_mv_unique
  ON mv_vendas_categoria(category_name);

✅ Antes: 120ms | Depois: 8ms → 15x mais rápido nas consultas com filtro

✅ Quando usar Materialized Views

Use MVs quando:

  • Consultas são complexas e lentas (muitos JOINs, agregações)
  • Volume de leitura >> volume de escrita
  • Dados podem ter algumas horas/dias de atraso
  • Performance é crítica (dashboards, relatórios)
  • Dados base mudam pouco (catálogo, regiões)

❌ Quando NÃO usar Materialized Views

Evite MVs quando:

  • Dados precisam ser em tempo real (estoque, transações)
  • Tabelas base mudam constantemente
  • Espaço em disco é limitado
  • A query base é simples e já é rápida

🏪 Casos Reais de Uso

🏪 E-commerce

  • Dashboard de vendas diárias
  • Produtos mais vendidos
  • Performance por região

🏦 Finanças

  • Relatório de transações
  • Análise de risco consolidada
  • Dashboard de investimentos

📋 Resumo

  1. View Regular: apenas define a query, não armazena dados
  2. Materialized View: executa a query e guarda o resultado fisicamente
  3. Performance: MVs podem ser 100-400x mais rápidas
  4. Trade-off: velocidade vs atualidade dos dados
  5. REFRESH: manual (bloqueante) ou CONCURRENTLY (não bloqueante)
  6. Índices: MVs podem ter índices próprios para acelerar consultas

Regra de ouro: Se pode esperar minutos/horas por dados atualizados e precisa de performance extrema → use MV!

🧪 Prática 2: Desempenho de Funcionários

Cenário: RH quer saber o desempenho de cada vendedor nos últimos 90 dias.

1 Crie a view regular primeiro:


CREATE OR REPLACE VIEW vw_desempenho_funcionarios AS
SELECT e.employee_id,
       e.first_name || ' ' || e.last_name AS nome,
       e.title AS cargo,
       COUNT(DISTINCT o.order_id) AS total_pedidos,
       SUM(od.quantity * od.unit_price) AS valor_total_vendido,
       ROUND(AVG(od.quantity * od.unit_price), 2) AS ticket_medio
FROM employees e
LEFT JOIN orders o ON e.employee_id = o.employee_id
LEFT JOIN order_details od ON o.order_id = od.order_id
GROUP BY e.employee_id, e.first_name, e.last_name, e.title;

⏱️ Execute com EXPLAIN ANALYZE e anote o tempo!

🧪 Prática 2: Criando a MV

2 Crie a Materialized View:


CREATE MATERIALIZED VIEW mv_desempenho_funcionarios AS
SELECT e.employee_id,
       e.first_name || ' ' || e.last_name AS nome,
       e.title AS cargo,
       COUNT(DISTINCT o.order_id) AS total_pedidos,
       SUM(od.quantity * od.unit_price) AS valor_total_vendido,
       ROUND(AVG(od.quantity * od.unit_price), 2) AS ticket_medio
FROM employees e
LEFT JOIN orders o ON e.employee_id = o.employee_id
LEFT JOIN order_details od ON o.order_id = od.order_id
GROUP BY e.employee_id, e.first_name, e.last_name, e.title
WITH DATA;

3 Verifique:

⏰ Agendamento com pg_cron


-- Instalar a extensão (se não tiver)
CREATE EXTENSION IF NOT EXISTS pg_cron;

-- Atualizar MV de vendas todo dia às 2h
SELECT cron.schedule(
  'refresh-mv-vendas',
  '0 2 * * *',
  $$REFRESH MATERIALIZED VIEW mv_vendas_categoria$$
);

-- Atualizar MV de funcionários a cada hora
SELECT cron.schedule(
  'refresh-mv-funcionarios',
  '0 * * * *',
  $$REFRESH MATERIALIZED VIEW CONCURRENTLY mv_desempenho_funcionarios$$
);

✅ Com pg_cron, o REFRESH é automático. Sem necessidade de scripts externos!

⏰ Alternativa: crontab do Linux


# Editar crontab
crontab -e

# Atualizar MV de vendas todo dia às 2h
0 2 * * * psql -d northwind -c "REFRESH MATERIALIZED VIEW mv_vendas_categoria;"

# Atualizar MV de funcionários a cada hora
0 * * * * psql -d northwind -c "REFRESH MATERIALIZED VIEW CONCURRENTLY mv_desempenho_funcionarios;"

✅ Boas Práticas

  • Nomeclatura: prefixo mv_ para distinguir de views regulares
  • Documente: frequência de refresh e tolerância a dados antigos
  • Índice único: sempre crie se pretende usar CONCURRENTLY
  • Monitore: tamanho das MVs e tempo de refresh
  • Plano B: saiba como fazer DROP e recriar se algo der errado

⚠️ Armadilhas Comuns

❌ Erros frequentes

  • Esquecer de dar REFRESH
  • Usar MV para dados em tempo real
  • Não criar índice único
  • MV muito grande sem particionamento

✅ Soluções

  • Agendar refresh automático
  • Usar view regular para dados críticos
  • Criar UNIQUE INDEX sempre
  • Particionar por período

🗑️ Removendo uma MV


-- Remover MV e todos os índices
DROP MATERIALIZED VIEW IF EXISTS mv_vendas_categoria;

-- Remover com CASCADE (se houver dependências)
DROP MATERIALIZED VIEW IF EXISTS mv_vendas_categoria CASCADE;

⚠️ CASCADE remove também objetos que dependem da MV. Use com cuidado!

🎓 Resumo Final

  1. View Regular = atalho, não guarda dados → sempre atual, mas lenta
  2. Materialized View = tabela física com resultados → rápida, mas precisa de REFRESH
  3. Ganho real: 100-400x mais rápido em queries complexas
  4. Trade-off: velocidade vs atualidade
  5. REFRESH: comum (bloqueante) ou CONCURRENTLY (não bloqueante)
  6. Índices: essenciais para performance e CONCURRENTLY
  7. Automação: pg_cron ou crontab para refresh agendado
