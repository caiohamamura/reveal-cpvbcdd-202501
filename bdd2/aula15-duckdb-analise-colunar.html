Hoje: e quando os dados estão em arquivos ou DataFrames e queremos analisar rapidamente com SQL?

Parte 1: Análise não é cadastro

Temos um CSV grande com vendas, logs, sensores, notas ou dados públicos.

Precisamos criar um servidor PostgreSQL, importar tudo e configurar tabelas antes de perguntar algo simples?

DuckDB entra como uma ponte: SQL conhecido, mas aplicado diretamente sobre arquivos e DataFrames.

OLTP: trabalho transacional

OLTP vem de Online Transaction Processing: processamento de transações online.

App Web / Mobile (Milhares de acessos)
↔
PostgreSQL (Transacional)
↔
Registros Singulares (Ex: Pedido #1234)

Características

Muitas operações pequenas, frequentes, concorrentes e com cuidado forte de integridade (Ex: Cadastro, Login, Compras).

OLAP: trabalho analítico

OLAP vem de Online Analytical Processing: processamento analítico online.

Data Lake / Arquivos (Milhões de registros)
→
(Motor Analítico)
→
Dashboard / BI (Visão Resumida)

Características

Percorre muitos registros filtrando e agrupando para devolver um resumo (Ex: Total de vendas por mês, Média de notas).

A diferença está na pergunta

Tipo Pergunta Operação dominante
OLTP Este pedido foi pago? Este usuário pode entrar? Ler ou alterar poucas linhas com consistência
OLAP Quanto vendemos por mês, categoria e cidade? Filtrar, agrupar e resumir muitas linhas

DuckDB foi desenhado principalmente para o segundo tipo de pergunta.

CSV e Parquet

CSV: texto linha a linha

data,produto,categoria,cidade,valor 2026-01-05,Mouse,Periferico,Capivari,50.0 2026-01-12,Teclado,Periferico,Elias Fausto,120.0

Fácil de abrir manualmente, mas precisa interpretar texto a cada leitura.

Parquet: arquivo colunar

schema: data: DATE produto: VARCHAR categoria: VARCHAR valor: DOUBLE

Guarda tipos e metadados; combina com consultas que leem poucas colunas.

Performance: texto contra colunas

CSV

  • precisa interpretar texto a cada leitura
  • lê a linha inteira
  • não guarda tipos fortes
  • compressão depende de arquivo externo

Parquet

  • guarda dados por coluna
  • permite ler só colunas usadas
  • guarda tipos e metadados
  • usa compressão eficiente por coluna

Em consultas analíticas, Parquet costuma reduzir leitura de disco, conversão de tipos e memória movimentada.

Parte 3: SQL sem servidor

DuckDB é um motor SQL analítico embarcado: roda dentro do processo da sua aplicação ou notebook.

  • não precisa criar servidor
  • não precisa usuário, senha e porta
  • consulta arquivos e DataFrames diretamente
  • usa SQL, filtros, agrupamentos e funções conhecidas

Preparar o ambiente

Para executar os exemplos em Python, instale o DuckDB no mesmo ambiente usado pelo notebook ou pelo VS Code.

python -m pip install duckdb pandas pyarrow

pandas cria DataFrames para os exemplos e pyarrow ajuda na leitura/escrita de Parquet.

SQL sobre DataFrame

import duckdb import pandas as pd df = pd.DataFrame({ "produto": ["Mouse", "Teclado", "Mouse", "Monitor"], "categoria": [ "Periferico", "Periferico", "Periferico", "Video" ], "valor": [50, 120, 70, 900], }) resultado = duckdb.sql(""" SELECT produto, SUM(valor) AS total FROM df GROUP BY produto ORDER BY total DESC """).df() resultado

Resultado esperado

produto total
0 Monitor 900.0
1 Mouse 120.0
2 Teclado 120.0

O ponto didático: a turma já sabe SQL. Agora o SQL também conversa com DataFrames.

Por que SQL aqui ajuda?

Com pandas

filtrado = df[ df["valor"] >= 100 ] por_categoria = ( filtrado .groupby("categoria") .agg( qtd=("produto", "count"), total=("valor", "sum"), ) .reset_index() ) resultado = por_categoria[ por_categoria["qtd"] >= 2 ]

Com SQL

SELECT categoria, COUNT(*) AS qtd, SUM(valor) AS total FROM df WHERE valor >= 100 GROUP BY categoria HAVING COUNT(*) >= 2 ORDER BY total DESC;

Filtro, agrupamento, agregação, critério sobre grupo e ordenação ficam juntos na mesma frase.

Um caso que fica imperativo em pandas

Queremos comparar cada venda com a média da sua categoria e manter apenas vendas acima dessa média.

pandas

medias = ( df.groupby("categoria")["valor"] .mean() .reset_index( name="media_categoria" ) ) com_media = df.merge(medias, on="categoria") resultado = com_media[ com_media["valor"] > com_media["media_categoria"] ]

SQL

SELECT produto, categoria, valor, AVG(valor) OVER ( PARTITION BY categoria ) AS media_categoria FROM df QUALIFY valor > media_categoria;

Legibilidade em consultas analíticas

pandas é excelente, mas muitas análises viram sequências de passos: filtra, agrupa, reseta índice, junta, filtra de novo.

SQL deixa a intenção mais declarativa: quais linhas entram, como agrupar, quais métricas calcular e quais grupos manter.

  • WHERE: filtra linhas antes do agrupamento
  • GROUP BY: define o nível do resumo
  • HAVING: filtra grupos depois da agregação
  • WINDOW: compara cada linha com estatísticas do seu grupo

    Parte 4: Arquivos como tabelas

    DuckDB permite consultar arquivos diretamente, sem etapa obrigatória de importação permanente.

    SELECT * FROM read_csv_auto('vendas.csv') LIMIT 10; 
    bdd2/

  vendas.parquet

    Agregando um CSV

    SELECT categoria, COUNT(*) AS qtd, SUM(valor) AS total FROM read_csv_auto('vendas.csv') GROUP BY categoria ORDER BY total DESC;
    categoria qtd total
    Computador 1 3200.0
    Video 2 1750.0
    Periferico 6 615.0
    Armazenamento 1 420.0
    Acessorio 2 130.0

    Consultando Parquet

    SELECT categoria, COUNT(*) AS qtd, SUM(valor) AS total FROM read_parquet('vendas.parquet') WHERE data >= DATE '2026-01-01' GROUP BY categoria ORDER BY total DESC;

    Parquet combina muito bem com consultas que leem muitas linhas, mas poucas colunas.

    Arquivos reais raramente vêm sozinhos

    Em pipelines de dados, é comum receber um arquivo por mês, por dia, por cidade ou por sistema.

    dados/vendas/
  ano=2026/
    mes=01/vendas_01.parquet
    mes=02/vendas_02.parquet
    mes=03/vendas_03.parquet
    mes=04/vendas_04.parquet

    DuckDB consegue consultar vários arquivos como se fossem uma tabela única.

    Glob: muitos arquivos, uma consulta

    SELECT cidade, DATE_TRUNC('month', data) AS mes, SUM(valor) AS total FROM read_parquet( 'dados/vendas/ano=2026/mes=*/vendas_*.parquet' ) WHERE categoria = 'Periferico' GROUP BY cidade, DATE_TRUNC('month', data) ORDER BY mes, total DESC;

    O padrão * permite abrir muitos arquivos sem escrever um loop em Python.

    Paralelo com big data

    Ideia Big data distribuido DuckDB local
    Dados particionados Pastas por ano, mes, cidade Ler glob de muitos arquivos
    Formato colunar Parquet em data lake read_parquet()
    Consulta declarativa SQL em engines como Spark/Trino SQL local no notebook
    Escala Cluster Uma máquina, sem servidor

    DuckDB não substitui um cluster, mas traz a mesma lógica de análise para dados que cabem numa máquina.

    Hive partitioning

    Esta convenção de pastas nasceu no Apache Hive (Hadoop) e virou padrão na indústria.

    dados/vendas/
  ano=2026/
    mes=01/
      vendas_01.parquet
    mes=02/
      vendas_02.parquet

    O DuckDB lê este padrão nativamente. As pastas ano=2026 e mes=01 viram colunas virtuais na consulta.

    Lendo partições Hive com DuckDB

    SELECT ano, mes, cidade, SUM(valor) AS total FROM read_parquet( 'dados/vendas/ano=*/mes=*/*.parquet', hive_partitioning = true ) WHERE ano = 2026 GROUP BY ano, mes, cidade ORDER BY mes, total DESC;

    DuckDB cria as colunas ano e mês a partir dos nomes das pastas.

    Também funciona com CSV

    SELECT ano, mes, categoria, COUNT(*) AS qtd FROM read_csv( 'dados/vendas_csv/ano=*/mes=*/*.csv', header = true, hive_partitioning = true ) WHERE categoria = 'Periferico' GROUP BY ano, mes, categoria ORDER BY ano, mes;

    Mesmo quando a origem ainda é CSV, o padrão de pastas já aproxima a organização de um data lake.

    Parte 5: Pipeline analítico

    Entrada CSV, Parquet, DataFrame
    →
    DuckDB SQL local no notebook
    →
    Saída resumo, CSV, Parquet

    O pipeline da aula usa os arquivos reais do repositório: ler, inspecionar, agregar e exportar.

    1. Criar tabela a partir do arquivo

    vendas.csv Disco local
    →
    Tabela vendas Memória (DuckDB)
    import duckdb con = duckdb.connect() con.sql(""" CREATE TABLE vendas AS SELECT * FROM read_csv_auto('vendas.csv') """)

    2. Inspecionar

    DESCRIBE vendas; SELECT COUNT(*) AS total_linhas FROM vendas; SELECT * FROM vendas LIMIT 10;

    O que buscar?

    • Colunas: quais nomes e campos existem?
    • Tipos: foram detectados corretamente?
    • Volume: quantas linhas temos no total?
    • Coerência: os dados fazem sentido visualmente?

    3. Agregar

    SELECT categoria, AVG(valor) AS media, SUM(valor) AS total FROM vendas GROUP BY categoria ORDER BY total DESC; SELECT produto, COUNT(*) AS quantidade FROM vendas GROUP BY produto ORDER BY quantidade DESC;

    Resumo (Exemplo)

    categoria média total
    Notebook 4500.00 90000.00
    Celular 2100.00 84000.00
    Periferico 150.00 15000.00

    4. Analisar no tempo

    SELECT DATE_TRUNC('month', data) AS mes, SUM(valor) AS total FROM vendas GROUP BY mes ORDER BY mes;

    Datas transformam registros soltos em série histórica: mês a mês, semana a semana, dia a dia.

    5. Exportar resultado

    Para CSV

    COPY ( SELECT categoria, SUM(valor) AS total FROM vendas GROUP BY categoria ) TO 'resumo_categorias.csv' (HEADER, DELIMITER ',');

    Para Parquet

    COPY ( SELECT categoria, SUM(valor) AS total FROM vendas GROUP BY categoria ) TO 'resumo_categorias.parquet' (FORMAT PARQUET);

    Parte 6: Integração direta (Postgres)

    O DuckDB possui uma extensão nativa para ler e escrever diretamente em bancos PostgreSQL.

    PostgreSQL Servidor remoto
    ↔
    DuckDB Análise local
    INSTALL postgres; LOAD postgres; ATTACH 'dbname=meu_banco user=postgres host=localhost' AS pg_db (TYPE postgres); -- Agora podemos consultar tabelas do Postgres diretamente SELECT * FROM pg_db.public.vendas LIMIT 5;

    Autenticação no PostgreSQL

    Na string de conexão

    -- Senha inline ATTACH 'dbname=vendas user=postgres host=localhost password=minha_senha' AS pg (TYPE postgres);

    Nunca versione senhas no código.

    Variáveis de ambiente

    # Variáveis padrão PostgreSQL $env:PGHOST="localhost" $env:PGPORT="5432" $env:PGDATABASE="vendas" $env:PGUSER="postgres" $env:PGPASSWORD="minha_senha"

    Basta usar ATTACH '' AS pg

    A extensão usa libpq internamente — todas as variáveis de ambiente padrão do PostgreSQL são suportadas, incluindo o arquivo ~/.pgpass.

    Opções de conexão

    String libpq

    -- Parâmetros básicos ATTACH 'dbname=vendas user=postgres host=127.0.0.1 port=5432' AS pg (TYPE postgres); -- Somente leitura ATTACH 'dbname=vendas user=postgres' AS pg (TYPE postgres, READ_ONLY);

    URI PostgreSQL

    ATTACH 'postgresql://postgres@localhost/vendas' AS pg (TYPE postgres); -- Ver tabelas disponíveis SHOW ALL TABLES; DETACH pg;

    A string de conexão segue o formato libpq do PostgreSQL: mesmos parâmetros usados em psql e psycopg2.

    Consultando tabelas do PostgreSQL

    Depois do ATTACH, as tabelas do Postgres ficam visíveis como tabelas DuckDB normais.

    ATTACH 'dbname=vendas user=postgres' AS pg (TYPE postgres); -- Consulta direta — dados lidos do Postgres em tempo real SELECT categoria, SUM(valor) AS total FROM pg.public.vendas GROUP BY categoria ORDER BY total DESC; -- Copiar tabela do Postgres para DuckDB local CREATE TABLE vendas_local AS FROM pg.public.vendas; DETACH pg;

    O DuckDB age como cliente SQL do Postgres: só trafega os dados necessários para responder a consulta.

    ETL direto entre fontes

    Parquet / CSV
    →
    Agrega (Em Memória)
    →
    PostgreSQL
    -- Consolidar milhares de Parquets e salvar o resumo no Postgres INSERT INTO pg.public.resumo_mensal SELECT DATE_TRUNC('month', data) AS mes, categoria, SUM(valor) AS total FROM read_parquet('dados/**/*.parquet') GROUP BY mes, categoria;

    O DuckDB atua como motor ETL: processa arquivos e envia o resumo pronto ao banco transacional, sem gerar arquivos temporários.

    COPY entre DuckDB e PostgreSQL

    Postgres → Parquet

    ATTACH 'dbname=vendas' AS pg (TYPE postgres, READ_ONLY); COPY pg.public.vendas TO 'backup/vendas.parquet' (FORMAT PARQUET);

    Parquet → Postgres

    ATTACH 'dbname=vendas' AS pg (TYPE postgres); COPY pg.public.vendas FROM 'dados/vendas_novas.parquet';

    COPY usa o protocolo binário do PostgreSQL — mais rápido que INSERT linha a linha para grandes volumes.

    Parte 8: Quando usar DuckDB?

    Pergunta PostgreSQL DuckDB
    Aplicação web com muitos usuários escrevendo? Sim Normalmente não
    Análise local de CSV e Parquet? Sim, com mais configuração Sim, muito bem
    Precisa de servidor? Normalmente sim Não
    Transações, permissões e restrições? Forte Não é o foco
    Exploração com agregações? Sim Sim, especialmente em arquivos
    Conectar diretamente ao PostgreSQL? Não aplicável Sim, com extensão postgres

    Mapa mental da disciplina

    PostgreSQL

    Relacional / Transacional. A fonte da verdade para apps.

    Redis

    Chave-Valor na memória. Cache e fila de alta velocidade.

    MongoDB

    Documentos flexíveis JSON. Catálogos e dados aninhados.

    Neo4j

    Grafos e Relacionamentos. Redes sociais e recomendações.

    DuckDB

    Analítico local (OLAP). Ponte SQL entre DataFrames, arquivos Parquet/CSV e bancos (ETL).

    O SQL é a linguagem universal que cruza essas ferramentas, seja para consultas simples (OLTP) ou analíticas (OLAP).
