    Fase 1: Por que Redis?

    Redis não é “MongoDB mais simples”. Ele resolve outro tipo de problema.

    Ideia central: Redis é um servidor de estruturas de dados em memória, usado junto com bancos como PostgreSQL ou MongoDB.

    Quando Redis aparece em sistemas reais?

    ProblemaRecurso do Redis
    Lembrar sessão de login por 30 minutosChave com TTL
    Contar visualizações em tempo realContador atômico
    Guardar carrinho temporárioHash + TTL na chave
    Processar tarefas em segundo planoLista como fila
    Registrar usuários online sem repetiçãoSet
    Ranking de jogoSorted set

    Comparação rápida

    BancoIdeia principalExemplo
    PostgreSQLDados relacionais, SQL, restriçõespedidos, alunos, vendas
    PostgreSQL JSONBRelacional com JSON flexívelatributos de produto
    MongoDBDocumentos como modelo principalperfis, catálogos
    RedisChave-valor e estruturas em memóriacache, sessões, filas
    Neo4jGrafo de relaçõesrecomendações

    Fase 2: Instalação local

    Vamos manter um ipython aberto e executar os comandos pelo cliente Python do Redis.

    Isso fica mais próximo de um backend real: a aplicação Python conversa com um servidor Redis.

    Windows: preparar ambiente

    winget install Python.Python.3.12 --accept-source-agreements --accept-package-agreements winget install Redis.Redis --accept-source-agreements --accept-package-agreements py -m pip install redis ipython

    No Windows, vamos usar o Redis instalado pelo winget. O Python acessa localhost:6379.

    Windows: testar servidor e abrir IPython

    redis-cli PING ipython

    Se aparecer PONG, deixe o ipython aberto para os próximos slides.

    Se o comando não responder, feche e abra o terminal ou inicie o serviço Redis no Windows.

    Linux: instalar com apt

    sudo apt update sudo apt install redis-server python3-redis ipython3 sudo systemctl enable --now redis-server ipython3

    No Linux, o Redis Server roda como serviço local e o IPython conversa com ele pela porta 6379.

    Fase 3: Python e chave-valor

    Antes de estruturas avançadas, vamos tratar Redis como um mapa: chave → valor.

    Conectar Python ao Redis

    import redis r = redis.Redis( host="localhost", port=6379, decode_responses=True, ) r.ping()

    Resposta esperada: True

    Comandos básicos

    r.set("nome", "Ana") r.get("nome") r.exists("nome") r.delete("nome") r.get("nome")

    set grava um valor

    get lê um valor

    exists verifica existência

    delete remove a chave

    Contadores atômicos

    r.set("visitas", 0) r.incr("visitas") r.incr("visitas") r.get("visitas")

    Pergunta: por que incr é melhor que ler, somar na aplicação e gravar de novo?

    Resposta esperada: porque a operação é atômica. Vários processos podem incrementar sem sobrescrever uns aos outros.

    Fase 4: Expiração e cache

    Uma chave que desaparece sozinha é uma ferramenta poderosa para sistemas web.

    TTL: tempo de vida da chave

    r.set("codigo:1234", "839201") r.expire("codigo:1234", 30) r.ttl("codigo:1234") r.get("codigo:1234")
    ResultadoSignificado
    TTL = número positivosegundos restantes
    TTL = -1a chave existe, mas não expira
    TTL = -2a chave não existe mais

    Mini-exercício: token de login

    Objetivo: criar uma sessão temporária que expira em 20 segundos.

    r.set("sessao:usuario:1", "logado") r.expire("sessao:usuario:1", 20) r.ttl("sessao:usuario:1") r.get("sessao:usuario:1") r.get("sessao:usuario:1") r.ttl("sessao:usuario:1")

    Depois das quatro primeiras chamadas, aguarde mais de 20 segundos e rode as duas últimas.

    Fase 5: Estruturas de dados

    Redis não guarda só strings. A força dele está nas estruturas prontas.

    String Hash List Set Sorted Set

    Hashes: objetos pequenos

    r.hset("aluno:1", "nome", "Ana") r.hset("aluno:1", "curso", "Informática") r.hset("aluno:1", "idade", 17) r.hget("aluno:1", "nome") r.hgetall("aluno:1") r.hincrby("aluno:1", "idade", 1) r.hgetall("aluno:1")
    SQLRedis Hash
    linhahash
    colunacampo
    chave primárianome da chave

    Hash não tem join, chave estrangeira nem SQL.

    Exercício: produto em hash

    Objetivo: representar um produto com nome, categoria, preço e estoque. Depois, reduzir o estoque.

    r.hset("produto:1", "nome", "Mouse") r.hset("produto:1", "categoria", "Periféricos") r.hset("produto:1", "preco", "59.90") r.hset("produto:1", "estoque", 20) r.hgetall("produto:1") r.hincrby("produto:1", "estoque", -1) r.hgetall("produto:1")

    Lists: filas e pilhas

    r.lpush("fila:emails", "email1") r.lpush("fila:emails", "email2") r.lpush("fila:emails", "email3") r.rpop("fila:emails") r.rpop("fila:emails") r.rpop("fila:emails")

    lpush insere à esquerda

    rpop remove da direita

    lpush + rpop cria comportamento de fila

    Uso real: pedidos de envio de e-mail, geração de relatórios, processamento assíncrono.

    Sets: valores únicos

    r.sadd("usuarios:online", "ana") r.sadd("usuarios:online", "bruno") r.sadd("usuarios:online", "ana") r.smembers("usuarios:online") r.scard("usuarios:online") r.sismember("usuarios:online", "ana") r.srem("usuarios:online", "ana")

    sadd adiciona ao conjunto

    smembers lista os valores

    scard conta quantos existem

    sismember testa se existe

    srem remove do conjunto

    Sets ignoram repetição: adicionar ana duas vezes mantém apenas uma ocorrência.

    Sorted sets: ranking com pontuação

    r.zadd( "jogo:ranking", { "ana": 500, "bruno": 700, "carla": 650, "diego": 300, "elisa": 900, }, ) r.zrevrange("jogo:ranking", 0, -1, withscores=True) r.zincrby("jogo:ranking", 100, "ana") r.zrevrange("jogo:ranking", 0, 2, withscores=True)

    zadd adiciona valores com pontuação

    zrevrange lista do maior para o menor

    0, -1 pega todos os elementos

    withscores=True mostra também a pontuação

    zincrby aumenta a pontuação de um item

    0, 2 pega os três primeiros

    Uso real: ranking de jogos, placar de alunos, posts mais curtidos, produtos mais acessados.

    Fase 6: Persistência e decisão técnica

    Redis é em memória, mas isso não significa “sempre temporário”.

    Redis perde tudo ao reiniciar?

    Redis guarda dados em memória para velocidade, mas pode salvar em disco conforme a configuração.

    Modos de persistência

    ModoIdeiaUso típico
    Sem persistênciadados desaparecem após reiníciocache puro
    RDBsnapshots periódicosbackup com perda aceitável
    AOFlog de operações de escritamaior durabilidade
    RDB + AOFmodo híbridoprodução com durabilidade

    Ativar AOF e RDB pelo IPython

    r.config_set("appendonly", "yes") r.config_set("appendfsync", "everysec") r.config_set("save", "900 1 300 10 60 10000") r.config_rewrite()

    appendonly yes: liga o AOF.

    appendfsync everysec: grava o log a cada segundo.

    save: liga snapshots RDB automáticos.

    config_rewrite: salva a configuração no arquivo do Redis.

    Verificar a configuração

    r.config_get("appendonly") r.config_get("appendfsync") r.config_get("save")

    Se aparecer yes, everysec e a regra 900 1 300 10 60 10000, AOF e RDB estão ativos.

    Como ler 900 1 300 10 60 10000?

    O Redis lê essa configuração em pares: segundos mudanças.

    ParSignificado
    900 1se pelo menos 1 chave mudou em 900 segundos, salve um snapshot RDB
    300 10se pelo menos 10 chaves mudaram em 300 segundos, salve um snapshot RDB
    60 10000se pelo menos 10000 chaves mudaram em 60 segundos, salve um snapshot RDB

    Ideia central: quanto mais mudanças acontecem, menor pode ser o intervalo até o próximo snapshot.

    Testar persistência na prática

    No IPython:

    r.set("persistencia:teste", "sobreviveu") r.save() r.get("persistencia:teste")

    No Windows, use PowerShell como administrador.

    Restart-Service Redis sudo systemctl restart redis-server

    Depois do reinício, rode r.get("persistencia:teste") de novo. O valor deve continuar existindo.

    Resumo da Aula

  • Redis é chave-valor, mas também é servidor de estruturas de dados
  • TTL torna sessões, tokens e caches mais simples
  • Hashes, lists, sets e sorted sets modelam problemas comuns
  • Redis pode persistir, mas não substitui automaticamente PostgreSQL
  • A decisão técnica começa pelo caso de uso, não pelo hype
