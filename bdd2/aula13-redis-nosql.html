    Fase 1: Por que Redis?

    Redis não é “MongoDB mais simples”. Ele resolve outro tipo de problema.

    Ideia central: Redis é um servidor de estruturas de dados em memória, usado junto com bancos como PostgreSQL ou MongoDB.

    Quando Redis aparece em sistemas reais?

    ProblemaRecurso do Redis
    Lembrar sessão de login por 30 minutosChave com TTL
    Contar visualizações em tempo realContador atômico
    Guardar carrinho temporárioHash + TTL na chave
    Processar tarefas em segundo planoLista como fila
    Registrar usuários online sem repetiçãoSet
    Ranking de jogoSorted set

    Comparação rápida

    BancoIdeia principalExemplo
    PostgreSQLDados relacionais, SQL, restriçõespedidos, alunos, vendas
    PostgreSQL JSONBRelacional com JSON flexívelatributos de produto
    MongoDBDocumentos como modelo principalperfis, catálogos
    RedisChave-valor e estruturas em memóriacache, sessões, filas
    Neo4jGrafo de relaçõesrecomendações

    Fase 4: Expiração e cache

    Uma chave que desaparece sozinha é uma ferramenta poderosa para sistemas web.

    TTL: tempo de vida da chave

    ResultadoSignificado
    TTL = número positivosegundos restantes
    TTL = -1a chave existe, mas não expira
    TTL = -2a chave não existe mais

    Mini-exercício: token de login

    Objetivo: criar uma sessão temporária que expira em 20 segundos.

    Depois das quatro primeiras chamadas, aguarde mais de 20 segundos e rode as duas últimas.

    Cache não é o dado principal

    sequenceDiagram
  participant App as Aplicação
  participant R as Redis
  participant DB as PostgreSQL
  App->>R: GET produto:10
  alt cache hit
    R-->>App: JSON do produto
  else cache miss
    App->>DB: SELECT produto
    DB-->>App: registro
    App->>R: SET produto:10 EX 300
  end

    Redis acelera a leitura, mas a fonte de verdade pode continuar sendo PostgreSQL.

    Fase 5: Estruturas de dados

    Redis não guarda só strings. A força dele está nas estruturas prontas.

    String Hash List Set Sorted Set

    Hashes: objetos pequenos

    SQLRedis Hash
    linhahash
    colunacampo
    chave primárianome da chave

    Hash não tem join, chave estrangeira nem SQL.

    Exercício: produto em hash

    Objetivo: representar um produto com nome, categoria, preço e estoque. Depois, reduzir o estoque.

    Lists: filas e pilhas

    lpush insere à esquerda

    rpop remove da direita

    lpush + rpop cria comportamento de fila

    Uso real: pedidos de envio de e-mail, geração de relatórios, processamento assíncrono.

    Sets: valores únicos

    Sets são úteis quando repetição não faz sentido: usuários online, alunos que entregaram uma tarefa, tags únicas.

    Sorted sets: ranking com pontuação

    Fase 6: Persistência e decisão técnica

    Redis é em memória, mas isso não significa “sempre temporário”.

    Redis perde tudo ao reiniciar?

    Explicação ruim:

    Redis é em memória, então perde tudo.

    Explicação melhor:

    Redis guarda dados em memória para velocidade, mas pode salvar em disco conforme a configuração.

    Modos de persistência

    ModoIdeiaUso típico
    Sem persistênciadados desaparecem após reiníciocache puro
    RDBsnapshots periódicosbackup com perda aceitável
    AOFlog de operações de escritamaior durabilidade
    RDB + AOFmodo híbridoprodução com durabilidade

    Resumo da Aula

  • Redis é chave-valor, mas também é servidor de estruturas de dados
  • TTL torna sessões, tokens e caches mais simples
  • Hashes, lists, sets e sorted sets modelam problemas comuns
  • Redis pode persistir, mas não substitui automaticamente PostgreSQL
  • A decisão técnica começa pelo caso de uso, não pelo hype
