    BSON e _id

    JSON

    • Formato textual
    • Ótimo para APIs e leitura humana
    • Tipos limitados

    BSON

    • Formato binário do MongoDB
    • Inclui tipos como ObjectId e datas
    • O campo _id identifica cada documento

    A estrutura parece JSON, mas o banco armazena e consulta BSON.

    Parte 2: CRUD e Consultas

    Inserir, buscar, projetar, atualizar e remover documentos

    Inserindo Documentos

    db.produtos.insertOne({ sku: "AC-100", nome: "Mouse Vertical", categoria: "acessorio", preco: 189.9, estoque: 45, specs: { conexao: "Bluetooth", dpi: 3200 }, tags: ["ergonomia", "escritorio"] })

    A coleção não exigiu CREATE TABLE. Isso é produtivo, mas também permite inconsistência se a aplicação não controlar o formato.

    Primeiras Consultas

    db.produtos.find() db.produtos.find({ categoria: "notebook" }) db.produtos.find( { categoria: "notebook" }, { nome: 1, preco: 1, estoque: 1 } )

    O segundo argumento é a projeção: quais campos aparecem no resultado.

    Filtros com Operadores

    db.produtos.find({ preco: { $gte: 1000, $lte: 6000 } }) db.produtos.find({ categoria: { $in: ["notebook", "celular"] } }) db.produtos.find({ $or: [ { estoque: { $lt: 10 } }, { tags: "promocao" } ] })

    Campos Aninhados

    A dot notation consulta campos dentro de objetos:

    db.produtos.find({ "specs.ram_gb": { $gte: 16 } }) db.produtos.find({ "specs.camera_mp": { $gte: 48 } })

    Esse é um dos motivos de MongoDB parecer natural para objetos da aplicação.

    Arrays

    db.produtos.find({ tags: "promocao" }) db.produtos.find({ tags: { $all: ["trabalho", "portatil"] } }) db.produtos.find({ tags: { $size: 2 } })

    Se um campo é array, uma busca por valor simples encontra documentos onde o array contém aquele valor.

    Atualizações

    db.produtos.updateOne( { sku: "NB-001" }, { $set: { preco: 4999.9 } } ) db.produtos.updateOne( { sku: "NB-001" }, { $inc: { estoque: -1 } } ) db.produtos.updateOne( { sku: "NB-001" }, { $push: { tags: "oferta-relampago" } } )

    Cuidado com Updates Amplos

    Filtro vazio altera muitos documentos:

    db.produtos.updateMany( {}, { $set: { revisado: true } } )

    Antes de atualizar ou remover, execute o filtro com find() e confirme o conjunto afetado.

    Remoção

    db.produtos.deleteOne({ sku: "AC-100" }) db.produtos.deleteMany({ categoria: "acessorio", estoque: 0 })

    A regra prática é a mesma do SQL: remova com filtros específicos e verificáveis.

    Flexibilidade não é ausência de projeto

    Agora a pergunta muda:

    Como um desenvolvedor de ADS decide se o modelo documental ficou bom?

    Problema-guia: marketplace simples

    Você está construindo o backend de um marketplace com produtos de categorias diferentes.

    notebook processador, RAM, SSD, tela, sistema operacional

    livro autor, ISBN, páginas, editora, idioma

    celular armazenamento, câmera, bateria, rede, sistema

    acessório compatibilidade, conexão, cor, material

    Desafio: o time de produto muda os atributos a cada sprint. Você quer velocidade sem criar bagunça.

    Atividade 1: inserir documentos diferentes

    use bdd2_aula12 db.produtos.insertMany([ { sku: "NB-ADS-01", nome: "Notebook ADS 14", categoria: "notebook", preco: 4200, specs: { ram_gb: 16, ssd_gb: 512, tela_polegadas: 14 }, tags: ["estudo", "programacao"] }, { sku: "LV-ADS-01", nome: "Livro Banco de Dados", categoria: "livro", preco: 180, specs: { autor: "Silberschatz", paginas: 1376, idioma: "pt-BR" }, tags: ["teoria", "sql"] } ])

    A vantagem aparece imediatamente: a coleção aceita formatos diferentes sem migration.

    Mas flexibilidade não é ausência de projeto

    Bom uso

    • campos variáveis por categoria
    • documentos parecidos, mas não idênticos
    • consultas frequentes conhecidas
    • validação mínima combinada entre app e banco

    Uso perigoso

    • cada documento tem um formato imprevisível
    • tipos mudam: número vira texto
    • a aplicação precisa adivinhar o schema
    • relatórios passam a quebrar silenciosamente

    Atividade 2: quebre o banco de propósito

    Insira um produto com tipos e estrutura ruins:

    db.produtos.insertOne({ sku: "BAD-001", nome: "Produto Bagunçado", categoria: "notebook", preco: "barato", specs: "depende do fornecedor", tags: "promocao" })
    db.produtos.find({ preco: { $gte: 1000 } }) db.produtos.aggregate([ { $group: { _id: "$categoria", precoMedio: { $avg: "$preco" } } } ])

    Discussão rápida

    O MongoDB errou ao aceitar esse documento?

  • Não necessariamente: flexibilidade é parte da proposta.
  • O erro foi não definir um contrato mínimo para a coleção.
  • Em sistemas reais, a validação pode estar no frontend, backend, banco ou nos três.
    •

    Validação com JSON Schema

    MongoDB também permite impor regras na coleção.

    db.createCollection("produtos_validados", { validator: { $jsonSchema: { bsonType: "object", required: ["sku", "nome", "categoria", "preco"], properties: { sku: { bsonType: "string" }, nome: { bsonType: "string" }, categoria: { enum: ["notebook", "livro", "celular", "acessorio"] }, preco: { bsonType: ["double", "int", "long", "decimal"], minimum: 0 }, specs: { bsonType: "object" }, tags: { bsonType: "array" } } } } })

    Teste a validação

    Deve falhar

    db.produtos_validados.insertOne({ sku: "BAD-002", nome: "Produto Inválido", categoria: "notebook", preco: "barato", specs: "texto" })

    Deve passar

    db.produtos_validados.insertOne({ sku: "OK-001", nome: "Notebook Validado", categoria: "notebook", preco: 3999.9, specs: { ram_gb: 16 }, tags: ["ads"] })

    Onde Validar? Camadas de Contrato

    CamadaResponsabilidadeExemplo
    FrontendExperiência e erro amigávelcampo preço aceita apenas número
    BackendRegra de negóciopreço promocional não pode ser maior que original
    BancoIntegridade mínima persistentesku, categoria e preço devem existir

    Flexibilidade boa é flexibilidade com contrato explícito.

    Modelagem: embed ou referência?

    Pedido e cliente: como você modelaria?

    Documento com cliente embutido

    { numero: "PED-001", cliente: { nome: "Ana", email: "ana@email.com" }, itens: [...] }

    Documento com referência

    { numero: "PED-001", cliente_id: ObjectId("..."), itens: [...] }

    Regra prática: modele pelo acesso

    SituaçãoTende a favorecerPor quê?
    Dados são lidos sempre juntosEmbeduma consulta resolve a tela
    Histórico deve congelar o valorEmbedpedido guarda nome/endereço da época
    Dado muda e deve refletir em todos os lugaresReferênciaevita duplicação inconsistente
    Relação muitos-para-muitos cresce muitoReferênciaevita documentos enormes

    Decisão de Modelagem: Embed ou Referência?

    Cenário: uma loja precisa guardar pedidos com cliente, endereço, itens, preço pago e status de entrega.

    O que você embute no documento do pedido? O que você referencia?

    • Quais dados devem ficar juntos no mesmo documento?
    • Quais campos devem ser apenas uma referência?

    Parte 4: Índices e Plano de Execução

    Consultas flexíveis também precisam de estrutura

    Índice como hipótese de uso

    Antes de criar índice, escreva a consulta-alvo:

    db.produtos.find({ categoria: "notebook", preco: { $lte: 6000 } }).sort({ preco: 1 })

    Qual índice você criaria para essa consulta?

    Consulta Sem Índice

    db.produtos.find({ categoria: "notebook", preco: { $lte: 6000 } }).explain("executionStats")

    Procure por COLLSCAN: o banco examinou a coleção sequencialmente.

    Criando Índice

    db.produtos.createIndex({ categoria: 1, preco: 1 }) db.produtos.find({ categoria: "notebook", preco: { $lte: 6000 } }).explain("executionStats")

    Agora procure por IXSCAN: a busca usou índice.

    Compare antes e depois

    Sem índice

    • procure COLLSCAN
    • muitos documentos examinados
    • bom para coleção pequena, ruim em produção

    Com índice

    • procure IXSCAN
    • menos documentos examinados
    • escritas ficam um pouco mais caras

    Índice em Campo Aninhado

    db.produtos.createIndex({ "specs.ram_gb": 1 }) db.produtos.find({ "specs.ram_gb": { $gte: 16 } })

    MongoDB indexa campos aninhados, mas o índice só ajuda se a consulta combina com o padrão de acesso.

    Índice Não É Decoração

    DecisãoBenefícioCusto
    Índice simplesAcelera filtro comumOcupa espaço e pesa escrita
    Índice compostoAjuda filtros combinadosOrdem dos campos importa
    Índice em campo aninhadoAcelera dot notationDepende da estabilidade do formato
    Muitos índicesMais caminhos de buscaInserções e updates ficam mais caros

    Aggregation: comece pela pergunta de negócio

    Evite "usar aggregate porque sim". Primeiro formule a pergunta:

    Quais categorias têm maior preço médio entre produtos disponíveis em estoque?

    db.produtos.aggregate([ { $match: { estoque: { $gt: 0 } } }, { $group: { _id: "$categoria", totalProdutos: { $sum: 1 }, precoMedio: { $avg: "$preco" } } }, { $sort: { precoMedio: -1 } } ])

    Preço Médio por Categoria

    db.produtos.aggregate([ { $match: { estoque: { $gt: 0 } } }, { $group: { _id: "$categoria", total: { $sum: 1 }, precoMedio: { $avg: "$preco" } } }, { $sort: { precoMedio: -1 } } ])

    Projetando o Resultado

    db.produtos.aggregate([ { $match: { categoria: "notebook" } }, { $project: { _id: 0, nome: 1, preco: 1, ram: "$specs.ram_gb", ssd: "$specs.ssd_gb" } }, { $sort: { ram: -1, preco: 1 } } ])

    Agrupando Pedidos

    db.pedidos.aggregate([ { $unwind: "$itens" }, { $group: { _id: "$itens.sku", unidades: { $sum: "$itens.qtd" }, receita: { $sum: { $multiply: ["$itens.qtd", "$itens.preco_unit"] } } } }, { $sort: { receita: -1 } } ])

    $unwind transforma cada item do array em uma entrada separada no pipeline.

    Parte 6: Da Coleção à API

    Expondo documentos MongoDB via HTTP

    API REST Minimalista

    Com Express e o driver nativo, uma coleção inteira em 10 linhas:

    const express = require('express'); const { MongoClient } = require('mongodb'); const app = express(); app.get('/produtos', async (req, res) => { const client = await MongoClient.connect('mongodb://localhost:27017'); const docs = await client.db('bdd2_aula12') .collection('produtos').find().toArray(); res.json(docs); }); app.listen(3000);

    Você já sabe a query. Express apenas a embrulha em um endpoint HTTP.

    API REST Auto-gerada

    O pacote express-restify-mongoose gera CRUD completo a partir de um modelo:

    const express = require('express'); const mongoose = require('mongoose'); const restify = require('express-restify-mongoose'); const app = express(); const ProdutoSchema = new mongoose.Schema({ sku: String, nome: String, preco: Number, categoria: String, estoque: Number }); const Produto = mongoose.model('Produto', ProdutoSchema); restify.serve(app, Produto); // Gera: GET /Produtos, POST /Produtos, // PUT /Produtos/:id, DELETE /Produtos/:id app.listen(3000);

    Testando a API no Navegador

    Com o servidor rodando em http://localhost:3000:

    curl http://localhost:3000/api/v1/Produtos # Ou abra no navegador: # http://localhost:3000/api/v1/Produtos?categoria=notebook

    O pacote suporta filtros, paginação, ordenação e projeção via query string — tudo mapeado para queries MongoDB.

    MongoDB no Mundo Real

    Quando usar, quando migrar, e o que empresas reais fizeram

    Onde MongoDB Brilha

  • Prototipagem rápida com schema em evolução
  • Dados que mapeiam 1:1 com objetos da aplicação
  • Escalabilidade horizontal (sharding) é obrigatória
  • Geolocalização, busca textual, time-series
  • Catálogos, feeds, logs, sessões, caches
    •

    MongoDB elimina o impedance mismatch entre código e banco quando os dados são naturalmente documentais.

    Onde Fica Mais Difícil

  • Transações ACID entre múltiplos documentos (possível, mas complexo)
  • Joins e relacionamentos complexos
  • Enforcement de schema estrito no banco
  • Relatórios e analytics com SQL-style aggregations
  • Consistência forte em sistemas financeiros
    •

    Schema flexível acelera o início, mas transferência de complexidade para a aplicação cobra o preço depois.

    Quando MongoDB faz sentido?

    Bom sinalPor quê?
    Dados naturalmente documentaisa aplicação carrega o agregado inteiro
    Schema muda com frequênciamenos migrations no início do produto
    Atributos variam por tipoevita dezenas de colunas nulas
    Time aceita modelar no appparte da complexidade sai do banco relacional

    Quando tomar cuidado?

    Sinal de alertaRisco
    Muitos relacionamentos complexoso modelo vira pseudo-relacional sem SQL
    Relatórios ad hoc frequentesSQL/BI pode ser mais produtivo
    Integridade forte é centralduplicação e referência precisam de disciplina
    Equipe acha que "sem schema" é sem regradívida técnica aparece rápido

    O Padrão de Migração

    graph LR
A["Startup com MongoDB"] --> B["Produto amadurece"]

C --> D["Transações necessárias"]
D --> E["Migra dados críticos para PostgreSQL"]
E --> F["MongoDB continua em workloads específicos"]

    MongoDB não desaparece — fica escopado aos casos onde documentos ainda vencem.

    Casos Reais de Migração

    EmpresaO que aconteceu
    Uber Começou com MongoDB para corridas; migrou transações para SQL e manteve MongoDB para telemetria/logs
    Netflix Usa MongoDB para catálogo e recomendações; pagamentos ficam em bancos relacionais
    PayPal Iniciou com MongoDB; migrou núcleo transacional para PostgreSQL por ACID rigoroso
    eBay MongoDB para catálogo e sessões; PostgreSQL para inventário e pedidos

    O padrão comum: polyglot persistence — usar o banco certo para o problema certo.

    Fechamento

    MongoDB não é só sintaxe diferente.

    É uma troca: mais ergonomia para documentos e mudanças rápidas, mais responsabilidade de modelagem para a aplicação e para o time.

    A pergunta final não é "MongoDB é melhor?", mas "este problema parece documental o bastante para justificar essa troca?".

    Lição da Aula

  • MongoDB acelera o desenvolvimento inicial com schema flexível
  • Índices e aggregation pipeline mantêm performance em escala
  • Para APIs, Express + Mongoose é o caminho padrão da indústria
  • Bancos não são religião: a escolha técnica deve evoluir com o produto
