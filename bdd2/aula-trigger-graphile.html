PARTE 1

Triggers — Uso com Cautela

Retomada: Functions no PostgreSQL

Na aula anterior, aprendemos que FUNCTIONs são blocos nomeados que:

  • Retornam valores ou executam operações
  • São chamadas explicitamente via SELECT ou PERFORM
  • Nos dão controle total sobre quando e como executam

Hoje: O que acontece quando a function é chamada automaticamente? 🤔

O que é um Trigger?

Um trigger é uma função que é executada automaticamente quando:

  • Um INSERT ocorre numa tabela
  • Um UPDATE modifica dados
  • Um DELETE remove registros

BEFORE ou AFTER de cada evento

Trigger em Ação: Atualização Automática


-- Tabelas
CREATE TABLE produtos (
  id SERIAL PRIMARY KEY,
  nome TEXT,
  estoque INTEGER DEFAULT 0
);

CREATE TABLE pedidos (
  id SERIAL PRIMARY KEY,
  produto_id INTEGER REFERENCES produtos(id),
  quantidade INTEGER
);

-- Trigger: atualiza estoque ao inserir pedido
CREATE TRIGGER trg_atualiza_estoque
  AFTER INSERT ON pedidos
  FOR EACH ROW
  EXECUTE FUNCTION atualizar_estoque();

-- Função do trigger
CREATE FUNCTION atualizar_estoque()
RETURNS TRIGGER AS $$
BEGIN
  UPDATE produtos
  SET estoque = estoque - NEW.quantidade
  WHERE id = NEW.produto_id;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

⚠️ Armadilhas dos Triggers

Efeitos Colaterais

Triggers podem modificar dados em tabelas que você não esperava

Obscuros

Difícil rastrear quando algo acontece "por trás"

Não Previsíveis

Performance pode degradar de formas inesperadas

Debugging Difícil

Erros em triggers são hard to trace back to source

Procedures Explícitas vs Triggers

Aspecto Procedure Explícita Trigger
Controle ✅ TOTAL — você chama quando quer ❌ Automático — firing pode surpreender
Rastreamento ✅ Simples — grep no código ⚠️ Complexo — depende do estado do DB
Performance ✅ Previsível ❌ Variável conforme carga
Quando usar Validações, operações controladas Auditoria leve, constraints complexas

💡 Regra: Prefira procedures explícitas para lógica de negócio

🎯 Exercício: Identificando o Problema

Cenário: Sistema de biblioteca com função emprestar_livro(id_livro, id_membro)

  • Insere registro em emprestimos
  • Atualiza disponivel = false em livros
  • Trigger trg_log_emprestimo insere em log_auditoria

⚠️ Problema: A função do trigger executa um SELECT pesado com JOIN em livros + autores para "detalhes do livro" — isso roda para cada empréstimo, causando lentidão extrema.

Pergunta: Onde está o problema? O que deveria ser feito?

✅ Resposta do Exercício

O problema: O trigger executa SELECTs pesados desnecessariamente

  • O trigger deve ser leve e atômico
  • Informações já estão em emprestimos — não precisa buscar novamente
  • Para logs detalhados: usar procedure explícita chamada após a transação
  • Ou usar NOTIFY para desacoplar

📝 Regra prática: Se o trigger precisa de dados, passe-os como NEW — não haga SELECT!

Resumo: Boas Práticas com Triggers

✅ Faça

  • Mantenha triggers simples
  • Use para auditoria leve
  • Valide dados com constraints

❌ Evite

  • SELECTs pesados em triggers
  • Lógica complexa de negócio
  • Triggers em cascata

PARTE 2

PostGraphile v5

API GraphQL a partir do PostgreSQL

O que é GraphQL?

GraphQL é uma linguagem de query para APIs:

  • Cliente especifica exatamente quais dados precisa
  • Um único endpoint para tudo
  • Retorna JSON estruturado

Exemplo de Query:


query {
  usuarios {
    id
    nome
    email
  }
}

GraphQL vs REST

Aspecto REST GraphQL
Endpoints Múltiplos: /users, /orders Um único: /graphql
Dados Retorna tudo ou nada Retorna exatamente o pedido
Overfetching ⚠️ Comum ✅ Não ocorre
Documentação Manual ou Swagger Introspecção nativa

Por que PostGraphile?

PostGraphile expõe seu schema PostgreSQL como API GraphQL automaticamente

  • 🔄 Zero código — tabelas viram tipos GraphQL automaticamente
  • 🔒 Segurança nativa — suporta RLS, JWT, permissões
  • ⚡ Performance — queries viram queries SQL otimizadas
  • 🛠️ Flexibilidade — expõe functions, views, procedures

📦 v5 já estável — grandes melhorias em performance e DX

GraphiQL: Explorando a API

GraphiQL é a ferramenta de exploração GraphQL embutida no PostGraphile

Acesse: http://localhost:5000/graphiql

✅ Benefícios

📝 Features

Criando o Schema api


-- 1. Criar schema dedicado
CREATE SCHEMA IF NOT EXISTS api;

-- 2. Habilitar extensão pgcrypto (para hash de senhas)
CREATE EXTENSION IF NOT EXISTS pgcrypto SCHEMA public;

-- 3. Tabelas dentro do schema api
CREATE TABLE api.usuarios (
  id SERIAL PRIMARY KEY,
  nome TEXT NOT NULL,
  email TEXT UNIQUE NOT NULL,
  senha_hash TEXT NOT NULL
);

CREATE TABLE api.pedidos (
  id SERIAL PRIMARY KEY,
  usuario_id INTEGER REFERENCES api.usuarios(id),
  produto TEXT NOT NULL,
  quantidade INTEGER NOT NULL,
  data_pedido TIMESTAMPTZ DEFAULT NOW()
);

Views e Functions Expostas

Por que não expor tabelas diretamente?

  • Views permitem seleção de colunas visíveis
  • Functions podem ter lógica de negócio
  • Você controla o que o cliente pode ver e fazer

-- View: expõe apenas dados necessários
CREATE VIEW api.meus_pedidos AS
SELECT
  p.id,
  p.produto,
  p.quantidade,
  p.data_pedido,
  p.usuario_id
FROM api.pedidos p;

-- Function: operação controlada
CREATE FUNCTION api.criar_pedido(
  p_usuario_id INTEGER,
  p_produto TEXT,
  p_quantidade INTEGER
) RETURNS api.pedidos AS $$
  INSERT INTO api.pedidos (usuario_id, produto, quantidade)
  VALUES (p_usuario_id, p_produto, p_quantidade)
  RETURNING *;
$$ LANGUAGE plpgsql SECURITY DEFINER;

Segurança: pgcrypto para Hash de Senhas

⚠️ NUNCA almacene senhas em texto puro ou com hash genérico!


-- Hash bcrypt via pgcrypto
INSERT INTO api.usuarios (nome, email, senha_hash) VALUES
  ('Ana', 'ana@email.com', crypt('senha123', gen_salt('bf'))),
  ('Bruno', 'bruno@email.com', crypt('senha456', gen_salt('bf')));

-- Verificação de senha
CREATE FUNCTION api.login(email TEXT, senha TEXT)
RETURNS TEXT AS $$
DECLARE
  usuario RECORD;
BEGIN
  SELECT * INTO usuario
  FROM api.usuarios
  WHERE email = login.email;

  IF usuario IS NULL THEN
    RAISE EXCEPTION 'Usuário não encontrado';
  END IF;

  -- Compara hash
  IF crypt(login.senha, usuario.senha_hash) = usuario.senha_hash THEN
    RETURN api.gerar_token(usuario.id);
  ELSE
    RAISE EXCEPTION 'Senha incorreta';
  END IF;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

JWT: Autenticação com Tokens


-- Geração simplificada de JWT
CREATE FUNCTION api.gerar_token(usuario_id INTEGER)
RETURNS TEXT AS $$
DECLARE
  header_base64 TEXT;
  payload_base64 TEXT;
  signature_base64 TEXT;
  signing_input TEXT;
  secret TEXT := 'segredo123';
BEGIN
  header_base64 := encode(
    to_jsonb('{"alg":"HS256","typ":"JWT"}'::json)::text,
    'base64url'
  );
  payload_base64 := encode(
    to_jsonb(json_build_object(
      'iat', extract(epoch from now())::INTEGER,
      'user_id', usuario_id
    ))::text,
    'base64url'
  );
  signing_input := header_base64 || '.' || payload_base64;
  signature_base64 := encode(
    hmac(signing_input, secret, 'sha256'),
    'base64url'
  );
  RETURN signing_input || '.' || signature_base64;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

📝 Em produção, use extensão pgjwt para JWT completo

RLS: Row Level Security

Isolamento por usuário — cada um vê apenas seus dados


-- Habilitar RLS na tabela
ALTER TABLE api.pedidos ENABLE ROW LEVEL SECURITY;

-- Política: usuário vê apenas seus próprios pedidos
CREATE POLICY usuarios_pedidos ON api.pedidos
  FOR ALL
  USING (
    usuario_id = NULLIF(
      current_setting('request.jwt.claim.user_id', true),
      ''
    )::INTEGER
  );

-- Para testar manualmente:
SET request.jwt.claim.user_id = '1';
SELECT * FROM api.pedidos;  -- só pedidos do usuário 1

RLS: Debugging com app.current_user_id

Dica: Use app.current_user_id via set_config para debugging


-- =============================================
-- DEBUGGING: app.current_user_id
-- =============================================

-- Simula usuário logado (para testar)
SET app.current_user_id = '1';
SELECT * FROM api.pedidos;  -- deve mostrar só usuário 1

-- Ou via PostGraphile com JWT:
SELECT set_config('request.jwt.claim.user_id', '1', false);

-- Resetar
RESET app.current_user_id;
-- =============================================

✅ Melhor que request.jwt.claim.user_id — mais fácil de testar

Mutations via Functions PostgreSQL


-- Mutation: criar pedido com validação
CREATE FUNCTION api.criar_pedido(
  p_produto TEXT,
  p_quantidade INTEGER
) RETURNS api.pedidos AS $$
DECLARE
  v_usuario_id INTEGER;
  v_resultado api.pedidos;
BEGIN
  -- Pegar user_id do JWT
  v_usuario_id := NULLIF(
    current_setting('request.jwt.claim.user_id', true),
    ''
  )::INTEGER;

  -- Inserir com validação
  IF p_quantidade <= 0 THEN
    RAISE EXCEPTION 'Quantidade deve ser maior que 0';
  END IF;

  INSERT INTO api.pedidos (usuario_id, produto, quantidade)
  VALUES (v_usuario_id, p_produto, p_quantidade)
  RETURNING * INTO v_resultado;

  RETURN v_resultado;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

Segurança: httpOnly Cookies vs localStorage

❌ localStorage

  • Vulnerável a XSS
  • JavaScript pode ler
  • Atacante injeta script → token roubado

✅ httpOnly Cookies

  • Inacessíveis ao JavaScript
  • Seguro contra XSS
  • PostGraphile suporta via --cookies

// localStorage (simplificado, menos seguro)
localStorage.setItem('token', jwt);

// httpOnly cookie (produção)
res.cookie('token', jwt, {
  httpOnly: true,
  secure: true,
  sameSite: 'strict'
});
