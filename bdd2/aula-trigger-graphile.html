    Parte 1: Triggers

    Execução automática — uso com cautela

    O que é um Trigger?

    Um trigger é uma função que executa automaticamente quando:

    • Um INSERT ocorre numa tabela
    • Um UPDATE modifica dados
    • Um DELETE remove registros

    ⏰ BEFORE = antes | AFTER = depois da operação

    BEFORE vs AFTER

    Aspecto BEFORE AFTER
    Quando executa Antes da operação Após a operação (committed)
    Pode modificar dados? ✅ SIM — pode alterar NEW ⚠️ Pode causar rollback
    Pode abortar? ✅ SIM — retorna NULL ⚠️ Sim — use exceções ainda fazem rollback!
    Use para Validação, normalização Auditoria, logs

    💡 BEFORE/AFTER + RAISE EXCEPTION aborta a operação!

    Variáveis: NEW, OLD e TG_OP

    NEW

    Dados novos

    • INSERT: valores inseridos
    • UPDATE: novos valores
    • DELETE: NULL

    OLD

    Dados antigos

    • INSERT: NULL
    • UPDATE: valores antigos
    • DELETE: sendo removidos

    TG_OP

    Operação atual

    • 'INSERT'
    • 'UPDATE'
    • 'DELETE'

    Exemplo: Trigger com TG_OP

    Por que RETURN é Necessário?

    ⚠️ O trigger é parte da chain — precisa retornar um valor!

    Comando BEFORE AFTER
    RETURN NEW Passa dados modificados Ignorado
    RETURN NULL Cancela a operação! Ignorado
    RAISE EXCEPTION Aborta transação Aborta transação

    Trigger em Ação: Estoque Automático

    1. Tabela produtos: armazena nome e estoque atual
    2. Tabela pedidos: cada pedido referencia um produto e uma quantidade
    3. Trigger: dispara AFTER INSERT em pedidos, para cada linha inserida
    4. Função: subtrai NEW.quantidade do estoque do produto referenciado
    5. RETURN NEW: convenção em triggers AFTER (valor é ignorado)

    Rollback dentro de Triggers

    ⚠️ Triggers executam dentro da mesma transação!

    ⚠️ Armadilhas dos Triggers

    Efeitos Colaterais

    Modificam dados em tabelas que você não esperava

    Obscuros

    Difícil rastrear quando algo acontece "por trás"

    Imprevisíveis

    Performance pode degradar inesperadamente

    Cascata

    Trigger dispara outro trigger → loop ou deadlock

    PostgreSQL limita: max_trigger_depth

    Triggers vs Procedures: Melhor Diretriz

    Não é "evite triggers" — é "use triggers com intenção"

    ⚡ Trigger

    Se esquecer vai quebrar dados

    Ex: auditoria, invariantes

    📦 Procedure

    Representa uma ação de negócio

    Ex: criar_pedido, emprestar_livro

    🔢 Function

    Computa sem efeitos colaterais

    Ex: calcular_desconto

    Procedure Explícita vs Trigger

    Aspecto Procedure Trigger
    Controle ✅ TOTAL — você chama ❌ Automático
    Rastreamento ✅ Simples — grep no código ⚠️ Complexo — estado do DB
    Performance ✅ Previsível ❌ Variável
    Quando usar Validações, operações controladas Auditoria leve, constraints complexas

    🧠 Ordem Correta de Decisão

    Antes de usar triggers ou procedures, pergunte:

    1. Posso usar CONSTRAINT? (FK, CHECK, UNIQUE)
    2. Posso não armazenar esse dado?
    3. É uma invariante que nunca pode quebrar? → TRIGGER
    4. É uma ação do sistema? → PROCEDURE

    Resumo: Boas Práticas com Triggers

    ✅ Faça

    • Mantenha triggers simples
    • Use para auditoria leve
    • Valide com BEFORE + RAISE
    • Retorne sempre um valor

    ❌ Evite

    • SELECTs pesados em triggers
    • Lógica complexa de negócio
    • Triggers em cascata sem limite
    • Depender de ordem de execução

    Parte 2: PostGraphile v5

    API GraphQL a partir do PostgreSQL

    O que é GraphQL?

    GraphQL é uma linguagem de query para APIs:

    • Cliente especifica exatamente quais dados precisa
    • Um único endpoint para tudo
    • Retorna JSON estruturado

    Exemplo de Query:

    GraphQL vs REST

    Aspecto REST GraphQL
    Endpoints Múltiplos: /users, /orders Um único: /graphql
    Dados Retorna tudo ou nada Retorna exatamente o pedido
    Overfetching ⚠️ Comum ✅ Não ocorre
    Documentação Manual ou Swagger Introspecção nativa

    Por que PostGraphile?

    PostGraphile expõe seu schema PostgreSQL como API GraphQL automaticamente

    • 🔄 Zero código — tabelas viram tipos GraphQL automaticamente
    • 🔒 Segurança nativa — suporta RLS, JWT, permissões (tópico avançado)
    • ⚡ Performance — queries viram SQL otimizadas
    • 🛠️ Flexibilidade — expõe functions, views, procedures

    📦 v5 já estável — grandes melhorias em performance e DX

    Por que Schema api Separado?

    ❌ Tables no schema public

    • Todas as tabelas expostas
    • Sem controle fino
    • Risco de segurança

    ✅ Schema api separado

    • Só expõe o que você quer
    • Views e functions controladas
    • Separação de concerns

    💡 O public é o padrão — mas o PostGraphile só expõe o que você especificar (--schema api)

    GraphiQL: Explorando a API

    GraphiQL é a ferramenta de exploração GraphQL embutida no PostGraphile

    Acesse: http://localhost:5000/graphiql

    ✅ Benefícios
