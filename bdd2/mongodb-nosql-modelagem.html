Agora a pergunta muda:

Como um desenvolvedor de ADS decide se o modelo documental ficou bom?

Problema-guia: marketplace simples

Você está construindo o backend de um marketplace com produtos de categorias diferentes.

notebook processador, RAM, SSD, tela, sistema operacional

livro autor, ISBN, páginas, editora, idioma

celular armazenamento, câmera, bateria, rede, sistema

acessório compatibilidade, conexão, cor, material

Desafio: o time de produto muda os atributos a cada sprint. Você quer velocidade sem criar bagunça.

Atividade 1: inserir documentos diferentes

use bdd2_aula12 db.produtos.insertMany([ { sku: "NB-ADS-01", nome: "Notebook ADS 14", categoria: "notebook", preco: 4200, specs: { ram_gb: 16, ssd_gb: 512, tela_polegadas: 14 }, tags: ["estudo", "programacao"] }, { sku: "LV-ADS-01", nome: "Livro Banco de Dados", categoria: "livro", preco: 180, specs: { autor: "Silberschatz", paginas: 1376, idioma: "pt-BR" }, tags: ["teoria", "sql"] } ])

A vantagem aparece imediatamente: a coleção aceita formatos diferentes sem migration.

Mas flexibilidade não é ausência de projeto

Bom uso

  • campos variáveis por categoria
  • documentos parecidos, mas não idênticos
  • consultas frequentes conhecidas
  • validação mínima combinada entre app e banco

Uso perigoso

  • cada documento tem um formato imprevisível
  • tipos mudam: número vira texto
  • a aplicação precisa adivinhar o schema
  • relatórios passam a quebrar silenciosamente

Atividade 2: quebre o banco de propósito

Insira um produto com tipos e estrutura ruins:

db.produtos.insertOne({ sku: "BAD-001", nome: "Produto Bagunçado", categoria: "notebook", preco: "barato", specs: "depende do fornecedor", tags: "promocao" })
db.produtos.find({ preco: { $gte: 1000 } }) db.produtos.aggregate([ { $group: { _id: "$categoria", precoMedio: { $avg: "$preco" } } } ])

Discussão rápida

O MongoDB errou ao aceitar esse documento?

  • Não necessariamente: flexibilidade é parte da proposta.
  • O erro foi não definir um contrato mínimo para a coleção.
  • Em sistemas reais, a validação pode estar no frontend, backend, banco ou nos três.
    •

    Validação com JSON Schema

    MongoDB também permite impor regras na coleção.

    db.createCollection("produtos_validados", { validator: { $jsonSchema: { bsonType: "object", required: ["sku", "nome", "categoria", "preco"], properties: { sku: { bsonType: "string" }, nome: { bsonType: "string" }, categoria: { enum: ["notebook", "livro", "celular", "acessorio"] }, preco: { bsonType: ["double", "int", "long", "decimal"], minimum: 0 }, specs: { bsonType: "object" }, tags: { bsonType: "array" } } } } })

    Teste a validação

    Deve falhar

    db.produtos_validados.insertOne({ sku: "BAD-002", nome: "Produto Inválido", categoria: "notebook", preco: "barato", specs: "texto" })

    Deve passar

    db.produtos_validados.insertOne({ sku: "OK-001", nome: "Notebook Validado", categoria: "notebook", preco: 3999.9, specs: { ram_gb: 16 }, tags: ["ads"] })

    Contrato mínimo para ADS

    CamadaResponsabilidadeExemplo
    FrontendExperiência e erro amigávelcampo preço aceita apenas número
    BackendRegra de negóciopreço promocional não pode ser maior que preço original
    BancoIntegridade mínima persistentesku, categoria e preço devem existir

    Flexibilidade boa é flexibilidade com contrato explícito.

    Modelagem: embed ou referência?

    Pedido e cliente: como você modelaria?

    Documento com cliente embutido

    { numero: "PED-001", cliente: { nome: "Ana", email: "ana@email.com" }, itens: [...] }

    Documento com referência

    { numero: "PED-001", cliente_id: ObjectId("..."), itens: [...] }

    Regra prática: modele pelo acesso

    SituaçãoTende a favorecerPor quê?
    Dados são lidos sempre juntosEmbeduma consulta resolve a tela
    Histórico deve congelar o valorEmbedpedido guarda nome/endereço da época
    Dado muda e deve refletir em todos os lugaresReferênciaevita duplicação inconsistente
    Relação muitos-para-muitos cresce muitoReferênciaevita documentos enormes

    Índice como hipótese de uso

    Antes de criar índice, escreva a consulta-alvo:

    db.produtos.find({ categoria: "notebook", preco: { $lte: 6000 } }).sort({ preco: 1 })

    Qual índice você criaria para essa consulta?

    Índice proposto

    db.produtos.createIndex({ categoria: 1, preco: 1 }) db.produtos.find({ categoria: "notebook", preco: { $lte: 6000 } }).sort({ preco: 1 }).explain("executionStats")

    A ideia é alinhar o índice ao padrão de acesso: filtro por categoria e faixa/ordenação por preço.

    Compare antes e depois

    Sem índice

    • procure COLLSCAN
    • muitos documentos examinados
    • bom para coleção pequena, ruim em produção

    Com índice

    • procure IXSCAN
    • menos documentos examinados
    • escritas ficam um pouco mais caras

    Aggregation: comece pela pergunta de negócio

    Evite “usar aggregate porque sim”. Primeiro formule a pergunta:

    Quais categorias têm maior preço médio entre produtos disponíveis em estoque?

    db.produtos.aggregate([ { $match: { estoque: { $gt: 0 } } }, { $group: { _id: "$categoria", totalProdutos: { $sum: 1 }, precoMedio: { $avg: "$preco" } } }, { $sort: { precoMedio: -1 } } ])

    Desafio final da aula

    Contexto: o gerente quer uma promoção para produtos caros, com estoque baixo e tags relacionadas a estudo ou trabalho.

    Monte uma consulta ou pipeline que retorne candidatos à promoção.

    db.produtos.find({ preco: { $gte: 1000 }, estoque: { $lte: 10 }, tags: { $in: ["estudo", "trabalho", "programacao"] } }, { nome: 1, categoria: 1, preco: 1, estoque: 1, tags: 1 })

    Quando MongoDB faz sentido?

    Bom sinalPor quê?
    Dados naturalmente documentaisa aplicação carrega o agregado inteiro
    Schema muda com frequênciamenos migrations no início do produto
    Atributos variam por tipoevita dezenas de colunas nulas
    Time aceita modelar no appparte da complexidade sai do banco relacional

    Quando tomar cuidado?

    Sinal de alertaRisco
    Muitos relacionamentos complexoso modelo vira pseudo-relacional sem SQL
    Relatórios ad hoc frequentesSQL/BI pode ser mais produtivo
    Integridade forte é centralduplicação e referência precisam de disciplina
    Equipe acha que “sem schema” é sem regradívida técnica aparece rápido

    Fechamento

    MongoDB não é só sintaxe diferente.

    É uma troca: mais ergonomia para documentos e mudanças rápidas, mais responsabilidade de modelagem para a aplicação e para o time.

    A pergunta final não é “MongoDB é melhor?”, mas “este problema parece documental o bastante para justificar essa troca?”.
