    Resolver problemas reais que SGBD-R tradicionais não tratavam bem

    O Manifesto NoSQL (2009)

    Movimento impulsionado por empresas web em escala massiva:

    Problemas que queriam resolver:

    • Escalabilidade horizontal simples
    • Esquema flexível / sem esquema
    • Alta disponibilidade (AP no CAP)
    • Desenvolvimento ágil sem migrations

    Bancos NoSQL populares:

    • Documento: MongoDB, CouchDB
    • Chave-valor: Redis, DynamoDB
    • Coluna larga: Cassandra, HBase
    • Grafo: Neo4j, ArangoDB

    JSONB: O Gigante "Desconhecido"

    Existe desde o PostgreSQL 9.4 — lançado em dezembro de 2014

    Linha do Tempo JSON no PostgreSQL

    VersãoAnoFeature
    9.22012Tipo json (texto, sem indexação eficiente)
    9.42014Tipo jsonb — binário, indexável
    9.52016jsonb_set(), jsonb_pretty()
    102017Operador ?? para jsonpath
    122019SQL/JSON Path (jsonb_path_query)
    142021Subscripting col['key']
    162023Mais operadores, melhor performance

    json vs jsonb: json armazena texto puro. jsonb armazena binário parseado — mais rápido para queries, suporta indexação.

    JSONB: O Que É?

    É um tipo de coluna que armazena JSON em formato binário:

    • Parseado na inserção (sem re-parse na query)
    • Suporta indexação via GIN
    • Operadores nativos para navegar
    • SQL/JSON Path para queries complexas

    Não é um "blob de texto"!

    • Tipo nativo do PostgreSQL
    • Validação automática (JSON inválido = erro)
    • Tipos suportados: string, number, boolean, null, array, object
    • Aninhamento ilimitado

    O "Pulo do Gato" do PostgreSQL

    Com JSONB você pode fazer em um único SELECT:

    
-- JOIN tabela relacional + filtro em JSONB + agregação SQL
SELECT c.nome as categoria,
       COUNT(*) as total,
       AVG(p.preco) as preco_medio,
       jsonb_object_agg(
         p.nome, p.atributos->>'processador'
       ) as processadores
FROM produtos p
JOIN categorias c ON p.categoria_id = c.id
WHERE p.atributos @> '{"ram_gb": 16}'::jsonb
GROUP BY c.nome;

    Isso no MongoDB? Precisa de $lookup + $match + $group — mais complexo e sem otimizações de query planner.

    Trabalhando com JSONB

    Operadores, funções e indexação na prática

    Operadores de Navegação

    OperadorRetornoExemplo
    -> JSONB atributos->'ram_gb' → 18
    ->> TEXT atributos->>'processador' → "M3 Pro"
    #> JSONB (path) atributos#>'{ports,0}' → "USB-C"
    #>> TEXT (path) atributos#>>'{ports,0}' → "USB-C"

    Regra: -> retorna JSONB (encadeável). ->> retorna TEXT (para comparações e WHERE).

    Operadores de Comparação / Contenção

    OperadorSignificadoExemplo
    @> Contém (esquerda contém direita) atributos @> '{"ram_gb":18}'
    <@ Contido em '{"ram_gb":18}' <@ atributos
    ? Chave existe? atributos ? 'isbn'
    ?| Alguma chave existe? atributos ?| array['isbn','autor']
    ?& Todas as chaves existem? atributos ?& array['isbn','autor']

    @> é o operador mais importante — é o que o índice GIN otimiza!

    Queries na Prática

    
-- 1. Filtrar por campo aninhado (retorna TEXT)
SELECT nome, atributos->>'processador' as cpu
FROM produtos
WHERE atributos->>'processador' = 'M3 Pro';

-- 2. Filtrar por número (cast necessário!)
SELECT nome, atributos->>'ram_gb' as ram
FROM produtos
WHERE (atributos->>'ram_gb')::int >= 16;

-- 3. Contenção — notebooks com SSD de 512GB
SELECT nome, preco
FROM produtos
WHERE atributos @> '{"ssd_gb": 512}'::jsonb;

-- 4. Chave existe — livros com ISBN
SELECT nome, atributos->>'isbn' as isbn
FROM produtos
WHERE atributos ? 'isbn';

    Indexação JSONB: GIN

    Sem índice, toda query JSONB = full table scan

    Sem Índice (Full Scan)

    
-- Sem índice: PostgreSQL escaneia TODAS as linhas
EXPLAIN ANALYZE
SELECT * FROM produtos
WHERE atributos @> '{"ram_gb": 16}'::jsonb;
          
    
Seq Scan on produtos  (cost=0.00..185.00 rows=1 width=200)
  Filter: (atributos @> '{"ram_gb": 16}'::jsonb)
  Rows Removed by Filter: 99999
Planning Time: 0.15 ms
Execution Time: 45.23 ms

    45ms para 100K linhas. Escala linearmente — em 10M linhas seriam 4.5 segundos.

    Índice GIN (Default)

    
-- GIN default: suporta @>, ?, ?|, ?&
CREATE INDEX idx_produtos_atributos_gin
  ON produtos USING GIN (atributos);

-- Mesma query, agora com índice
EXPLAIN ANALYZE
SELECT * FROM produtos
WHERE atributos @> '{"ram_gb": 16}'::jsonb;
          
    
Bitmap Index Scan on idx_produtos_atributos_gin
  Index Cond: (atributos @> '{"ram_gb": 16}'::jsonb)
Planning Time: 0.25 ms
Execution Time: 0.35 ms

    0.35ms — 129x mais rápido!

    GIN com jsonb_path_ops (Mais Leve)

    
-- jsonb_path_ops: menor, mais rápido para @>
-- MAS só funciona com @> (não suporta ?, ?|, ?&)
CREATE INDEX idx_produtos_atributos_path
  ON produtos USING GIN (atributos jsonb_path_ops);

    GIN default

    • Suporta @>, ?, ?|, ?&
    • Índice maior
    • Mais versátil

    jsonb_path_ops

    • Só @> (contenção)
    • Índice ~3x menor
    • Mais rápido para @>

    Índice em Campo Específico

    
-- Para queries com ->> (retorna TEXT), use B-tree ou GIN
-- B-tree: ideal para =, <, >, BETWEEN
CREATE INDEX idx_produtos_ram
  ON produtos ((atributos->>'ram_gb'));

-- Agora range queries são rápidas!
EXPLAIN ANALYZE
SELECT nome, atributos->>'ram_gb' as ram
FROM produtos
WHERE (atributos->>'ram_gb')::int >= 16;

    Dica: Se sempre filtra pelo mesmo campo JSONB, índice de expressão (B-tree) > GIN. Para campos variados, use GIN.

    SQL/JSON Path (PostgreSQL 12+)

    Queries complexas dentro de documentos — como XPath para JSON

    jsonb_path_query

    
-- Extrair array de ports
SELECT nome,
  jsonb_path_query(atributos, '$.ports[*]') as port
FROM produtos
WHERE categoria = 'notebook';

-- Filtrar por condição dentro do documento
SELECT nome, preco
FROM produtos
WHERE jsonb_path_exists(
  atributos,
  '$.ram_gb ? (@ >= 16)'
);

-- Busca recursiva: encontrar qualquer campo "ram_gb"
SELECT nome,
  jsonb_path_query(atributos, '$.**.ram_gb') as ram
FROM produtos;

    jsonb_path_query_array

    
-- Agregar resultados como array
SELECT nome,
  jsonb_path_query_array(
    atributos, '$.ports[*]'
  ) as todas_ports
FROM produtos
WHERE atributos ? 'ports';

-- Retorno:
-- nome         | todas_ports
-- MacBook Pro  | ["USB-C","HDMI","SD"]
          
    
-- Comparação com LIKE dentro do JSON
SELECT nome
FROM produtos
WHERE jsonb_path_exists(
  atributos,
  '$.** ? (@.string() like_regex "Pro" flag "i")'
);

    Computed Column para Filtros JSONB

    PostGraphile não filtra dentro de JSONB por padrão. Solução: computed columns.

    
-- Computed column: extrai ram_gb como integer
CREATE FUNCTION produtos_ram_gb(p produtos)
RETURNS INTEGER AS $$
  SELECT (p.atributos->>'ram_gb')::INTEGER;
$$ LANGUAGE sql STABLE;

-- Agora aparece no GraphQL!
          
    
query {
  allProdutos {
    nodes {
      nome
      ramGb  # ← computed column funciona no SELECT!
      preco
    }
  }
}

    Quando Usar O Quê?

    JSONB vs Tabelas Normalizadas vs MongoDB

    Use JSONB Quando...

    • Atributos variam por tipo — e-commerce, configs, metadata
    • Estrutura muda frequentemente — configs de app, feature flags
    • Dados são "acessórios" — o core é relacional, JSONB complementa
    • Precisa de SQL + documentos — melhor dos dois mundos
    • Already on PostgreSQL — sem infra nova, sem equipe nova

    Use Tabelas Normalizadas Quando...

    • Estrutura é estável — pouco muda, schema bem definido
    • Muitos relacionamentos — FK, JOINs, integridade referencial
    • Consultas complexas entre entidades — relações N:N
    • Dados financeiros / críticos — constraints rígidas protegem

    Use MongoDB Quando...

    • Precisa de sharding horizontal simples — petabytes de dados
    • Tudo é documento — não há relações significativas
    • Equipe já domina MongoDB — e não precisa de SQL
    • Change streams nativos — real-time pipelines

    Na prática: 90% dos casos que usam MongoDB seriam melhor servidos por PostgreSQL + JSONB.

    Guia de Decisão

    graph TD
  A["Precisa de esquema flexível?"] -->|Não| B["Tabelas normalizadas"]
  A -->|Sim| C["Precisa de relações + ACID?"]
  C -->|Não| D{"Dados em petabytes?"}
  D -->|Sim| E["MongoDB com sharding"]
  D -->|Não| F["PostgreSQL JSONB"]
  C -->|Sim| F

    Resumo da Aula

    1. SGBD-R tradicional sofre com esquemas dinâmicos e dados esparsos
    2. NoSQL resolveu flexibilidade mas sacrificou ACID, SQL e consistência
    3. JSONB existe desde 2014 — tipo nativo, binário, indexável
    4. Operadores: ->, ->>, @>, ?, #> para navegar e filtrar
    5. GIN index (default ou jsonb_path_ops) para queries rápidas
    6. SQL/JSON Path para queries complexas dentro de documentos
    7. PostGraphile expõe JSONB como GraphQL, com computed columns para filtros
    8. Na dúvida: use JSONB dentro do PostgreSQL — raramente precisa de MongoDB
