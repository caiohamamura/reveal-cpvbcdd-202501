Parte 1: Procedures e Funções

Rotinas armazenadas dentro do SGBD

Stored Procedures

O que é: sequência de comandos SQL armazenada e executada diretamente no banco

  • Centraliza regras e rotinas no SGBD
  • Pode receber parâmetros
  • Não retorna valor diretamente na consulta
  • Executada com CALL

Quando usar:

  • Automatizar rotinas com vários comandos
  • Calcular multas, atualizar saldos, registrar logs
  • Encapsular lógica operacional

Cuidado: uso excessivo pode aumentar o acoplamento ao SGBD

Funções

O que é: rotina que retorna um valor e pode ser usada em expressões SQL

  • Retorna um valor (diferente da procedure)
  • Pode ser usada dentro de SELECT
  • Calcula ou transforma valores
SELECT nome, calcular_idade(data_nasc) FROM funcionario;

A função calcular_idade recebe a data de nascimento e retorna a idade — exemplo de função definida pelo usuário.

Procedure × Função

Característica Procedure Função
Retorna valor? Não diretamente Sim, obrigatoriamente
Usada em SELECT? Não Sim
Recebe parâmetros? Sim Sim
Finalidade típica Automatizar rotinas Calcular/transformar valores
Invocação CALL nome() Dentro de expressões SQL

Parte 2: Triggers

Ações automáticas disparadas por eventos no banco

O que é uma Trigger?

Definição: rotina executada automaticamente em resposta a eventos como INSERT, UPDATE ou DELETE

Tipo Quando executa
BEFORE INSERT Antes de gravar o registro
AFTER INSERT Depois de gravar o registro
BEFORE UPDATE Antes da alteração
AFTER UPDATE Depois da alteração
BEFORE DELETE Antes da remoção
AFTER DELETE Depois da remoção

Caso clássico: registrar em historico_preco toda alteração de preço com OLD e NEW values

Cuidados com Triggers

⚠️ Riscos do uso excessivo:

  • Regras de negócio ficam menos visíveis para quem mantém o sistema
  • Impacto no desempenho em operações muito frequentes
  • Trigger que altera a mesma tabela pode causar recursão

✅ Bons usos:

  • Auditoria automática
  • Validações antes de inserção
  • Histórico de alterações
  • Manter campos calculados atualizados

Exemplo: Trigger de Auditoria

CREATE OR REPLACE FUNCTION fn_audit_preco() RETURNS TRIGGER AS $$ BEGIN INSERT INTO historico_preco( produto_id, preco_antigo, preco_novo, data_alteracao ) VALUES ( OLD.id, OLD.preco, NEW.preco, NOW() ); RETURN NEW; END; $$ LANGUAGE plpgsql; CREATE TRIGGER trg_audit_preco AFTER UPDATE OF preco ON produto FOR EACH ROW EXECUTE FUNCTION fn_audit_preco();

Parte 3: Views

Consultas armazenadas que simplificam o acesso a dados

Views — Conceitos

O que é: consulta armazenada que apresenta dados de uma ou mais tabelas

  • Armazena a definição da consulta, não os dados
  • Simplifica consultas complexas
  • Pode limitar colunas/linhas visíveis

Segurança: expõe apenas colunas/linhas autorizadas ao usuário

CREATE VIEW vw_funcionarios_publico AS SELECT nome, setor FROM funcionario;

CPF e salário ficam ocultos para quem consulta a view

Limitações de Views

  • Algumas views não permitem atualizações diretamente (JOINs, agregações)
  • Não armazenam dados — executam a consulta a cada acesso
  • Não melhoram desempenho por si só (a consulta roda a cada vez)

Pergunta comum na prova: view armazena dados? Não — armazena a definição da consulta.

Parte 4: Materialized Views

O resultado da consulta salvo fisicamente

View × Materialized View

Característica View comum Materialized View
Armazena dados? Não — só a definição Sim — resultado físico
Desempenho Executa a cada consulta Resultado pré-calculado (rápido)
Dados atualizados? Sempre em tempo real Pode ficar defasada
Ocupa espaço? Não Sim
Atualização Automática Manual (REFRESH)

Quando usar Materialized View?

✅ Adequada quando:

  • Consulta pesada executada com frequência
  • Dados não precisam estar em tempo real
  • Relatórios, dashboards, agregações

⚠️ Problema principal:

  • Se novas vendas forem inseridas após o último REFRESH, os totais ficam desatualizados
  • Evita recalcular, mas exige estratégia de atualização
 CREATE MATERIALIZED VIEW mv_vendas_mes AS SELECT DATE_TRUNC('month', data) AS mes, SUM(valor) AS total FROM vendas GROUP BY 1; REFRESH MATERIALIZED VIEW mv_vendas_mes;

Parte 5: NoSQL — Visão Geral

Modelos de dados além do relacional

Por que NoSQL?

  • Maior flexibilidade de estrutura
  • Melhor escalabilidade para certos cenários
  • Modelos diferentes do relacional para problemas diferentes

A escolha do modelo depende do tipo de consulta e da forma como os dados se relacionam — não da "moda" do momento.

Os 4 modelos NoSQL

Modelo Ideia Exemplo de uso Ferramenta
Chave-valor Par chave → valor Cache, sessões, contadores Redis
Documentos Documentos JSON flexíveis Catálogos, perfis, CMS MongoDB
Colunar Armazenamento por colunas Análise de grandes volumes DuckDB
Grafos Nós e relacionamentos Redes sociais, recomendações Neo4j

Cenário → Modelo

Questão clássica de prova: associar o cenário ao modelo mais adequado

Cenário Modelo
Sessão de usuário com expiração Chave-valor
Catálogo com produtos de estruturas variadas Documentos
Consulta analítica sobre grande volume de dados Colunar
Rede de relacionamentos entre pessoas Grafos

Parte 6: Chave-valor — Redis

Pares chave → valor em memória com estruturas de dados

Redis: conceitos-chave

  • Armazena pares chave → valor
  • Recuperação de dados pela chave
  • Estruturas: String, Hash, List, Set, Sorted Set
  • TTL: chaves com expiração automática

Usos comuns:

Cache de dados acessados com frequência

Sessões de login com expiração

Contadores atômicos em tempo real

Filas de tarefas assíncronas

Rankings ordenados por pontuação

Comandos essenciais

Comando Ação Estrutura
SET / GET Gravar / ler valor String
EXPIRE / TTL Definir / verificar expiração Qualquer
INCR Incremento atômico String (número)
HSET / HGETALL Campos de um objeto Hash
LPUSH / RPOP Fila (FIFO) List
SADD / SMEMBERS Conjunto sem repetição Set
ZADD / ZREVRANGE Ranking ordenado Sorted Set

Parte 7: Documentos — MongoDB

Dados semiestruturados em documentos JSON

MongoDB: conceitos-chave

  • Armazena documentos JSON (BSON)
  • Flexibilidade: cada documento pode ter campos diferentes
  • Dados aninhados (subdocumentos e arrays)
  • Sem necessidade de schema fixo

Ideal para: catálogos com atributos variáveis, perfis, conteúdo dinâmico

SQL MongoDB
Tabela Collection
Linha Documento
Coluna Campo
JOIN Embutir / $lookup

Embutir × Referenciar

✅ Embutir quando:

  • Dados lidos frequentemente juntos
  • Relação 1:poucos
  • Dados "pertencem" ao documento pai

⚠️ Problema de embutir demais:

  • Aumento de redundância
  • Dificuldade de atualização
  • Documento pode crescer demais

Consultas em Documentos

Filtros comuns no MongoDB:

  • Condições são passadas como objetos JSON
  • Operadores especiais começam com $
// Encontrar produtos com preço > 100 db.produtos.find({ preco: { $gt: 100 } })

Operadores de comparação

$eq = equal (igual a)

$gt = greater than (maior que)

$lt = less than (menor que)

$gte = greater than or equal to (maior ou igual a)

$lte = less than or equal to (menor ou igual a)

Parte 8: Colunar — DuckDB

Armazenamento orientado a colunas para análise de dados

Colunar: conceitos-chave

Ideia central: armazenar dados por colunas em vez de linhas

  • Favorece leitura seletiva de colunas
  • Melhor compressão dos dados
  • Ideal para consultas analíticas (OLAP)
  • Menos adequado para OLTP (escritas frequentes)

Quando é vantajoso:

  • Consulta acessa poucas colunas de muitas linhas
  • Grandes conjuntos de dados
  • Agregações: SUM, AVG, COUNT, GROUP BY

Linhas × Colunar: se a consulta precisa de todas as colunas de uma linha, o armazenamento por linhas é mais eficiente

Parte 9: Grafos — Neo4j

Dados como nós, relacionamentos e propriedades

Grafos: conceitos-chave

  • Dados representados como nós e relacionamentos
  • Cada nó e relacionamento pode ter propriedades
  • Consultas baseadas em travessia de relacionamentos
  • Linguagem: Cypher

Ideal para:

  • Redes sociais (amigos em comum)
  • Sistemas de recomendação
  • Rotas e caminhos entre pontos
  • Detecção de fraudes

Menos adequado: armazenar um código temporário acessado apenas por uma chave → chave-valor

Exemplo Cypher

MATCH (p:Pessoa {nome: "Ana"})-[:SEGUE]->(amigo) RETURN amigo.nome MATCH (a:Pessoa)-[:SEGUE*2]->(recomendacao) WHERE a.nome = "Ana" RETURN DISTINCT recomendacao.nome

Encontrar o menor caminho entre duas pessoas = travessia de relacionamentos
