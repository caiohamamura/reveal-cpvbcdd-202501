Etapa 1: Procedures e Funções

Dois recursos para centralizar lógica dentro do SGBD.

Ideia central: em vez de executar comandos um a um na aplicação, você empacota a lógica no banco.

O que é uma Procedure?

  • Conjunto de comandos SQL armazenados no SGBD
  • Pode ter parâmetros de entrada e saída
  • Não retorna valor diretamente — executa uma rotina
  • Ideal para: rotinas operacionais, fechamento de período, processamento em lote

Vantagem: centraliza regras e rotinas executadas diretamente no banco

Risco: uso excessivo aumenta o acoplamento da aplicação ao SGBD

O que é uma Função?

  • Retorna um valor e pode ser usada em expressões SQL
  • Pode receber parâmetros
  • Exemplos: calcular idade a partir da data de nascimento, formatar CPF
  • Pode ser chamada no SELECT, WHERE, ORDER BY

Uma função que recebe data_nascimento e retorna a idade é chamada de função definida pelo usuário (UDF).

Procedure × Função

Critério Procedure Função
Retorno Não retorna valor direto Retorna um valor
Uso em SELECT Não Sim
Parâmetros Sim Sim
Caso típico Rotinas e processamentos Cálculos e transformações
Chamada CALL proc() SELECT fn()

Procedures e funções podem receber parâmetros. Afirmar que nunca podem é errado.

Etapa 2: Triggers

Ações automáticas disparadas por eventos no banco.

Ideia central: uma trigger executa automaticamente em resposta a INSERT, UPDATE ou DELETE.

Tipos de Trigger

Tipo Quando dispara Uso típico
BEFORE INSERT Antes de gravar o registro Validar ou transformar dado antes de inserir
AFTER INSERT Depois de gravar Log de novo registro
BEFORE UPDATE Antes de alterar Impedir alterações indevidas
AFTER UPDATE Depois de alterar Registrar alteração em tabela de auditoria
AFTER DELETE Depois de excluir Arquivar o registro removido

Caso clássico: auditoria de preços

Tabela produto e tabela historico_preco.
Sempre que o preço mudar, registrar: preço anterior, preço novo e data.

Recurso mais adequado: Trigger AFTER UPDATE

Cuidados com Triggers

Risco 1 — Invisibilidade: regras de negócio escondidas em triggers ficam menos visíveis para quem mantém o sistema.

Risco 2 — Desempenho: triggers em operações muito frequentes podem degradar a performance.

Risco 3 — Recursão: uma trigger que altera a mesma tabela que a disparou pode causar recursão ou comportamento inesperado.

Etapa 3: Views

Consultas armazenadas que simplificam o acesso aos dados.

Ideia central: uma view é uma consulta salva que se comporta como tabela — ela não armazena fisicamente os dados (em geral).

Para que serve uma View?

Finalidade Exemplo
Simplificar consultas complexas Ocultar joins de 4 tabelas atrás de uma view vw_relatorio
Segurança e controle de acesso Expor apenas nome e setor do funcionário, ocultando CPF e salário
Camada de abstração Renomear colunas ou transformar dados sem alterar a tabela base

Limitação das Views

A view armazena a definição da consulta, não os dados.

A cada acesso, a consulta é executada novamente.

Limitação: algumas views não permitem UPDATE diretamente, especialmente se envolvem joins, agregações ou funções.

Caso real: empresa tem tabela de funcionários com salário, CPF e setor.
Usuário deve ver apenas nome e setor. → Criar uma view com apenas essas colunas.

Etapa 4: Materialized Views

Views que guardam fisicamente o resultado da consulta.

View × Materialized View

Critério View comum Materialized View
Armazena dados? Não (executa ao consultar) Sim (resultado gravado em disco)
Atualização Sempre atual (consulta ao vivo) Pode ficar desatualizada
Desempenho Depende da consulta Mais rápida para consultas pesadas
Uso típico Abstração e segurança Relatórios e dashboards periódicos

Quando usar Materialized View?

Ideal quando: consulta pesada executada com frequência e os dados não precisam estar sempre em tempo real.

Problema: se novas vendas forem inseridas após a última atualização, a materialized view pode apresentar valores desatualizados.

As três afirmações verdadeiras: views simplificam acesso, materialized views armazenam resultado, materialized views podem exigir atualização para refletir mudanças.

Etapa 5: NoSQL — Visão Geral

Diferentes problemas pedem diferentes modelos de banco.

Por que NoSQL?

  • Maior flexibilidade de estrutura
  • Escalabilidade horizontal
  • Modelos diferentes do relacional
  • Casos de uso que SQL resolve mal

Importante: a escolha entre modelos NoSQL deve considerar o tipo de consulta e a forma como os dados se relacionam, não o nome da ferramenta.

As 4 categorias NoSQL

Modelo Dados como Exemplo de ferramenta Caso típico
Chave-valor Pares chave → valor Redis Cache, sessões
Documentos Documentos JSON/BSON MongoDB Catálogo de produtos
Colunar Colunas agrupadas Cassandra, DuckDB Análise de grandes volumes
Grafos Nós e arestas Neo4j Redes sociais, rotas

Etapa 6: Chave-Valor

O modelo mais simples e mais rápido do NoSQL.

Dados armazenados como pares (chave, valor). A recuperação acontece pela chave.

Quando usar Chave-Valor?

Cenário Por quê funciona bem
Cache de dados acessados com frequência Acesso ultra-rápido por chave em memória
Sessões de usuário com expiração automática (TTL) Redis tem suporte nativo a expiração de chaves
Contadores atômicos (visitas, likes) INCR é atômico — sem race condition
Fila de tarefas simples Listas como LPUSH / RPOP

Não é adequado para: consultas complexas, joins, ou modelagem de relacionamentos.

Etapa 7: Banco de Documentos

Dados semiestruturados em documentos JSON.

Características de Bancos de Documentos

  • Armazenam dados como documentos (JSON, BSON)
  • Flexibilidade de estrutura — documentos podem ter campos diferentes
  • Suportam dados aninhados (listas, subdocumentos)
  • Sem obrigatoriedade de schema rígido

Ideal para: catálogo de produtos em que cada tipo tem atributos muito diferentes

Embedding: embutir dados relacionados no mesmo documento é vantajoso quando eles são frequentemente lidos juntos.

Embedding × Referência

Estratégia Quando usar Desvantagem
Embutir (embedding) Dados lidos juntos com frequência Redundância e dificuldade de atualização se repetido em muitos documentos
Referenciar Dados reutilizados em muitos documentos Múltiplas consultas para montar o dado

Etapa 8: Banco Colunar

Otimizado para leitura de grandes volumes analíticos.

Colunar × Orientado a Linhas

Linhas (OLTP): eficiente para ler/gravar uma linha completa — adequado para transações do dia a dia.

Colunas (OLAP): armazena os valores de cada coluna juntos — eficiente para ler poucas colunas de muitas linhas.

Vantagem colunar: compressão eficiente (valores similares na mesma coluna) e leitura seletiva — só lê as colunas que a consulta pede.

Não é adequado para: transações linha a linha, operações OLTP comuns.

Quando o colunar brilha?

Cenário Colunar?
Consulta analítica: SELECT AVG(salario) FROM empregados em 50 M de linhas ✔ Sim
Armazenar sessão temporária de usuário com TTL ✘ Não (chave-valor)
Recomendação baseada em conexões entre pessoas ✘ Não (grafos)
Análise de grandes conjuntos de dados com muitas leituras agregadas ✔ Sim

Etapa 9: Banco de Grafos

Modelo nativo para relacionamentos complexos.

O que é um banco de grafos?

  • Dados representados como nós, relacionamentos e propriedades
  • Consultas exploram caminhos e conexões
  • Travessia de relacionamentos é eficiente por design

Não é tabela, não é JSON, não é chave-valor.

Casos de uso de Grafos

Cenário Grafo?
Recomendação de amigos em rede social ✔ Ideal
Menor caminho entre duas pessoas ✔ Ideal — travessia de relacionamentos
Rotas entre pontos (GPS) ✔ Ideal
Análise de relacionamentos entre documentos ✔ Adequado
Armazenar um código temporário acessado apenas por uma chave ✘ Não — use chave-valor
