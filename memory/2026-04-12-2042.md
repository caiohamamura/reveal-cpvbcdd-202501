# 🎓 Roteiro de Slides: Programação Concorrente em IoT (45 min)

## **[Slide 1] Título e Introdução (5 min)**
*   **Título:** Relógios, NTP, Temporizadores e Osciladores
*   **Subtítulo:** Dominando o Tempo em Microcontroladores (ESP8266)
*   **Objetivos:** Listar os 3-4 objetivos chave (Ex: Entender `delay()` vs. `millis()`, Implementar concorrência).
*   **Engajamento Inicial:** Pergunta provocativa no slide: *Como um microcontrolador sabe quanto tempo passou?*

## **[Slide 2] O Coração da Máquina: O Clock (3 min)**
*   **Conceito:** O que é Clock? É a frequência que define a velocidade de execução das instruções.
*   **Visual:** Diagrama simples: Oscilador $\rightarrow$ Frequência (Hz) $\rightarrow$ Velocidade de Processamento.
*   **Exemplo:** ESP8266 $\approx$ 80MHz.
*   **Analogia:** Comparar com um motor: Quanto maior a frequência, mais rápido ele teoricamente pode operar.

## **[Slide 3] ⏳ O Paradoxo do `delay()` (5 min)**
*   **Código:** Mostrar o bloco `delay()` simples (piscar um LED).
*   **O Problema:** Usar `delay()` paralisa o programa. Se eu esperar 1 segundo, *nada mais* roda nesse tempo.
*   **Desafio Interativo:** Mostrar o desafio de piscar um LED **E** tocar um buzzer diferentes ao mesmo tempo. Mostrar o código falho.
*   **Conclusão:** `delay()` = Bloqueio total do sistema.

## **[Slide 4] 🚀 Conceito de Concorrência (5 min)**
*   **Verdade:** Microcontroladores não fazem "Multitarefa Real" (como um PC com vários núcleos).
*   **Simulação:** Fazem **Alternância Rápida** (Cooperativa). Eles trocam de tarefa tão rápido que *parece* simultâneo.
*   **Visual:** Um gráfico de Gantt simples mostrando o `loop()` alternando entre 3 funções (LED $\rightarrow$ Buzzer $\rightarrow$ Sensor).

## **[Slide 5] ⏳ Solução 1: `millis()` - A Base da Concorrência (7 min)**
*   **Conceito:** `millis()` mede os milissegundos decorridos desde o início.
*   **Mecânica:** Em vez de *esperar*, nós **verificamos** o tempo.
*   **Código:** Apresentar o trecho de código comparando `delay()` vs. `if (millis() - ultimoTempo > intervalo) { ... }`.
*   **Ação:** Demonstrar a evolução do código para adicionar o buzzer, mostrando que ambos os `if` rodam no mesmo ciclo do `loop()`.

## **[Slide 6] ⚙️ Solução 2: `Ticker.h` - Automatizando Tarefas (5 min)**
*   **Conceito:** Timers executam funções *automaticamente* em intervalos fixos, sem que o `loop()` precise checar constantemente.
*   **Vantagem:** Código mais limpo e organizado (separação de responsabilidades).
*   **Demonstração:** Mostrar o código com `timerLED.attach(1, piscar);`
*   **Ideal para:** Tarefas de fundo (background tasks) periódicas.

## **[Slide 7] ⚡️ Solução 3: Interrupts - O Evento Imediato (5 min)**
*   **Conceito:** O hardware "interrompe" o fluxo normal do processador para executar uma rotina crítica.
*   **Quando usar:** Eventos de tempo crítico (Botão Pressionado, Sinal Recebido).
*   **⚠️ ALERTA CRÍTICO:** **NUNCA** usar funções lentas (`Serial.print`, `delay`) dentro de uma função de *Interrupt*. Elas travam o sistema!

## **[Slide 8] 🔄 Comparativo Geral (3 min)**
*   **Visual:** Usar a tabela que criei: `delay()` $\leftarrow$ `millis()` $\leftarrow$ `Timer.h` $\leftarrow$ `Interrupt`
*   **Mensagem Chave:** O nível de complexidade vs. o nível de controle necessário.

## **[Slide 9] 🌐 Introdução ao NTP (O Problema do Tempo) (5 min)**
*   **Problema:** `millis()` só mede o *tempo decorrido*, não *que horas são*.
*   **Solução:** NTP (Network Time Protocol).
*   **Mecânica:** Pergunta à internet: "Que horas são?" (Exibir o código básico).

## **[Slide 10] ⚠️ CALLOUT BOX CRÍTICO: Confiabilidade do Tempo (Slide de Ênfase)**
*   **Título:** O Ponto Cego do NTP.
*   **Conteúdo:** O NTP depende de **Internet**. Se o WiFi cair, o tempo falha.
*   **A SOLUÇÃO DE OURO:** **RTC (Real Time Clock)**. Ele usa uma bateria dedicada para manter a hora local, mesmo sem rede. (Gancho perfeito para a próxima aula!)
